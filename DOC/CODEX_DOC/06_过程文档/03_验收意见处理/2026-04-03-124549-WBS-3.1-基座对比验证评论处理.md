# WBS 3.1 基座对比验证评论处理

## 评论来源

1. `#4 WBS 3.1 交互画布与渲染基座选型`
   - 评论时间：`2026-04-03T04:42:35Z`
   - 评论内容：希望 `WBS 3.1` 不只停留在分析结论，而是增加基座选型的对比验证；选择几个典型方案，构建验证页面，用户将依据验证页面效果决定最终方案。

## 技术判断

### 1. 该意见合理，且直接改变 `WBS 3.1` 的交付形态

1. 当前 `WBS 3.1` 已有设计正文，但交付物主要仍是分析型文档。
2. 对于 `DOM / SVG / Canvas` 这种强交互基座选型，仅靠文字描述不足以支撑最终拍板。
3. 用户明确提出“要看验证页面效果再决定”，说明该节点需要补充可观察、可操作的验证面，而不只是结论。

### 2. 本轮不应新开抽象父节点，应直接扩充 `WBS 3.1`

1. 用户评论明确指出“在当前节点 `WBS 3.1` 增加基座选型的对比验证”。
2. 因此本轮应直接扩充 `WBS 3.1` 的工作目标、成果物与验收方法，而不是再拆一个新的并列节点。
3. 扩充后的 `WBS 3.1` 应转为“分析文档 + 验证页面 + 测试用例”三件套。

### 3. 对比验证页应选取三类典型方案

1. `DOM + CSS`：
   - 代表“尽量保持纯 DOM / CSS 表达”的轻方案。
   - 便于观察线条表达、语义化和样式上限。
2. `DOM + SVG`：
   - 代表当前推荐的混合方案。
   - 便于验证节点端口、边路径、自动化可测性与表达平衡。
3. `Canvas`：
   - 代表位图主导方案。
   - 便于用户直观看到视觉密度与交互语义之间的取舍。

### 4. 验证页必须使用统一观察指标

1. 三种方案必须基于同一批节点和边。
2. 三种方案都应至少提供：
   - 节点拖拽
   - 画布平移
   - 缩放
   - 画布尺寸配置
   - 坐标或状态提示
3. 页面中需要明确展示每种方案对 `chrome mcp` / 自动化测试的影响说明，避免用户只看视觉不看工程后果。

## 执行决策

1. 新增本文件，记录本轮评论处理结论。
2. 更新 `WBS 3.1` 的研制计划文档：
   - 工作目标改为“分析 + 对比验证页”
   - 成果物补入前端验证页实现文件
   - 验收方法补入可执行测试与手工观察路径
3. 更新 `WBS 3.1` 设计正文：
   - 增加“对比验证页设计”“验证指标”“当前结论为暂定推荐”内容
4. 在前端实现独立的 `WBS 3.1 Base Lab` 验证页：
   - 不并入 `Knowledge / Graph / Tracks` 三主 tab
   - 作为单独验证入口存在
5. 为验证页补入前端测试，至少覆盖：
   - 入口存在
   - 三个方案卡存在
   - 可从主工作台切换到验证页
6. 本轮完成后同步 GitHub `#4` 正文，使远端契约与本地文档一致。

## 不执行项

1. 不把 `Canvas` 直接升级为当前主方案。
2. 不删除现有 `DOM + SVG` 路线，只把它从“分析推荐”提升为“经验证页验证的候选主方案”。
3. 不把验证页长期塞进产品主导航，避免污染正式产品信息架构。
