# WBS 4 四工作面任务树与原型空白导出分析

## 输入来源

1. GitHub issue `#12` 评论 `4188331135`
2. 评论来源级别：`Collaborator`
3. 处理结论：可直接作为本轮修改依据

## 原始意见整理

用户这条语音评论可以结构化理解为两类问题：

### 1. 任务树与计划口径错误

1. 当前 `WBS 4` 仍保留 `知识库 / 关系图 / 多轨编排` 这组三分法。
2. 用户真正要求的是把 `WBS 4` 直接改成四个工作面分支：
   - `卷宗`
   - `蓝图推演`
   - `剪辑编排`
   - `场景调度`
3. 因此，不只是改原型标题，而是要把 GitHub Project、GitHub Issues、本地研制计划三处的 `WBS 4` 计划一起重做。

### 2. 当前原型输出结果失真

1. 现有四张核心原型图内容密度过低。
2. 更严重的问题是导出的 `PNG` 主体区域几乎为空白，和 `.pen` 中设想的内容不一致。
3. 用户判断“还不如之前的截图”，说明当前产物已经不能承担设计沟通价值。

## 意见合理性分析

这条意见合理，原因如下：

1. `WBS 1.7.2` 已经把产品中区主工作面改成四个角度，如果 `WBS 4` 仍按旧三分法拆任务，计划和原型会自相矛盾。
2. `GitHub Project` 是用户读取大盘计划的主要入口；若这里仍保留旧三视图，用户自然会判断“没有理解要求”。
3. 原型图的职责是承接后续技术选型和实现边界；如果导出的 `PNG` 本身接近空白，则无论 `.pen` 文件里写了什么，都等于交付失败。

## 本轮处理目标

1. 把 `WBS 4` 从“旧三视图分支”改成“四工作面分支”。
2. 本地 `03_研制计划/`、相关设计说明、GitHub issue 契约、GitHub Project 树形结构同步改口。
3. 查清当前 `WBS 1.7.2` 原型导出为何接近空白，并修到输出结果可读。
4. 在不回退“四工作面”方向的前提下，补回每张图的主体内容密度。

## 本轮不做的事

1. 不在这一轮重新扩回八套路线。
2. 不把四工作面再退回 `知识库 / 关系图 / 轨道` 旧主导航。
3. 不在没有定位导出根因前直接盲改颜色或文案冒充修复。

## 拟执行方案

### 1. 计划层

1. 更新 `WBS 4` 父节点目标与后续节点定义。
2. 将 `WBS 4.1 / 4.2 / 4.3` 重定义为：
   - `WBS 4.1 卷宗工作面设计`
   - `WBS 4.2 蓝图推演工作面设计`
   - `WBS 4.3 剪辑编排工作面设计`
3. 新增：
   - `WBS 4.4 场景调度工作面设计`
4. 各工作面下继续保持：
   - `要素设计`
   - `能力验证`
   - `功能验证`
   - `样式验证`

### 2. 原型层

1. 先复现并定位 `PNG` 导出接近空白的根因。
2. 将“图像内容稀薄”和“导出链路失真”拆开处理。
3. 在修复导出链路后，再提升四张图的信息密度与结构表达。

### 3. 协同层

1. 完成本地文档修改后，同轮更新 GitHub issue 标题、body、评论回写和 Project 树结构。
2. 评论回写中明确说明：
   - 已按四工作面重排
   - 已处理原型空白导出问题
   - 当前验证结果是什么

## 当前风险

1. `WBS 4` 重排会影响已有 issue 编号、子节点说明和 Project 排序，需要整轮同步，不能只改一处。
2. 原型导出问题如果来自 `Pencil CLI` 能力边界，可能需要替换 `PNG` 导出链路，但仍保留 `.pen` 原稿。

## 验证方法

1. 人工核验 `WBS 4` 相关本地计划文档已改成四工作面口径。
2. 人工核验 GitHub issue / Project 树中不再保留 `知识库 / 关系图 / 多轨编排` 作为 `WBS 4` 子分支。
3. 人工核验 `DOC/CODEX_DOC/04_研发文档/03_原型图/WBS-1.7.2/` 下四张 `PNG` 已恢复主体内容，不再接近空白。
