# WBS 4.1 卷宗读写边界与事实层定位分析

## 评阅来源

- GitHub issue：`#44 WBS 4.1 卷宗工作面设计`
- 评论：`4190107945`

## 评论转译

评论的核心不是单纯要求“多几个按钮”，而是在追问以下设计问题：

1. 卷宗到底是不是可编辑界面。
2. 如果卷宗可以编辑，它应负责哪些编辑。
3. 如果卷宗不编辑，它为什么还能被称为工作面。
4. 当前 `BaseLab` 版卷宗为什么操作过少，无法体现关系、出场和事实连接的管理价值。

## 合理性判断

这条意见是合理的，原因有三点：

1. 当前 `WBS 4.1` 设计文档已经把卷宗定义为“对象事实管理工作面”，而不是只读浏览页，因此实现层如果只剩浏览和跳转，会与设计口径冲突。
2. 当前实现虽然有右侧检查器，但它主要停留在普通字段编辑，关系对象、引用字段和出场语境没有被充分表达，导致用户感知上仍像“能看一点、改一点”的半成品。
3. 用户提出的“人与人如何产生联系、在哪些场景发生过关系、标签和身份是否可改”本质上都属于事实层问题，确实应该先在卷宗中成立，而不是推给时间线或场景视图。

## 我的判断

卷宗最合理的定位不是下面两种极端：

1. 不是只读资料页。
2. 也不是吞掉其他视图职责的超级编辑器。

更合适的定义是：

`卷宗 = 对象事实层的读写混合工作面`

也就是：

1. 可以直接编辑对象字段。
2. 可以直接编辑引用字段。
3. 可以直接编辑独立关系对象。
4. 可以回查对象出场事件与章节切片投影。
5. 但不负责时间轨、空间舞台、蓝图推演和章节主编排。

## 本轮实施结论

基于上述判断，本轮不改 `WBS` 结构，仍然在 `WBS 4.1.2 / 4.1.3 / 4.1.4` 内补强 `BaseLab` 独立验证页，重点补：

1. 引用字段编辑入口
2. 关系对象档案与快速新建
3. 对象出场记录与章节切片投影回查
4. 设计说明中对卷宗读写边界的明确说明

## 不在本轮处理的内容

1. 主工作台融合
2. 轨道拖拽
3. 场景摆位
4. 蓝图推演联动

这些内容应继续留在 `WBS 4.1.5` 或其他视图节点中处理，避免卷宗工作面再次失焦。
