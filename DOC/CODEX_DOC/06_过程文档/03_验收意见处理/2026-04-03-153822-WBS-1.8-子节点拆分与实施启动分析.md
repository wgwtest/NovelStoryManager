# WBS 1.8 子节点拆分与实施启动分析

## 评论来源

1. GitHub issue：`#39 WBS 1.8 BaseLab验证工具构建方案`
2. 评论链接：`https://github.com/wgwtest/NovelStoryManager/issues/39#issuecomment-4182324844`
3. 评论原意：
   - 当前 `BaseLab` 工具方案可以接受
   - 希望判断是否需要继续细化 `WBS 1.8` 子节点
   - 如果意见一致，则拆分子任务并开展实施

## 现状判断

1. 该评论已经明确认可 `WBS 1.8` 的总体方案方向，因此继续停留在“只有一个方案节点”的状态没有必要。
2. 但当前仓库里仍缺少真正可执行的 `BaseLab` 实施节点，导致后续编码只能继续挂在 `#39` 之下，不利于树形理解和验收分层。
3. 同时，不适合把 `WBS 1.8` 继续细分成过多空节点：
   - 当前真正独立的工作对象并不多
   - 过度拆分会让 Project 树比代码实现更复杂
   - 后续新的实验类型应在真实出现时再补节点，而不是预先堆出很多占位项

## 结论

1. 该意见合理，应当拆分 `WBS 1.8` 子节点。
2. 当前阶段采用“最小但清晰”的两级实施拆分更合适：
   - `WBS 1.8.1 BaseLab独立应用底座`
   - `WBS 1.8.2 WBS 3.1实验迁移与入口解耦`
3. 不建议本轮继续拆出第三、第四个纯占位节点。后续只有在出现新的独立实验对象时，再补新的 `WBS 1.8.x` 子节点。

## 拆分理由

### WBS 1.8.1 BaseLab独立应用底座

承载的真实工作对象：

1. 新建 `apps/base-lab/` 独立应用
2. 建立实验入口首页与统一注册表
3. 接入 monorepo workspace、脚本、构建和测试
4. 明确 `BaseLab` 与主应用的共享边界

### WBS 1.8.2 WBS 3.1实验迁移与入口解耦

承载的真实工作对象：

1. 将现有 `WBS 3.1` 基座对比实验迁入 `BaseLab`
2. 从 `apps/web` 中移除 `BaseLab` 入口和页面耦合
3. 回写 `WBS 3.1` 的成果物、验收路径和说明文档

## 实施顺序判断

1. 必须先做 `WBS 1.8.1`，否则 `WBS 1.8.2` 没有稳定迁移目标。
2. `WBS 1.8.2` 可以紧接着进行，因为当前唯一已存在的实验就是 `WBS 3.1` 基座对比页。
3. 这两步都属于低耦合 `Prep`，不会破坏主工作台的正式业务建模方向。

## 本轮执行决定

1. 在本地 `03_研制计划/` 中新增 `WBS 1.8.1` 与 `WBS 1.8.2` 节点文档。
2. 同步创建 GitHub issues，并通过原生子 issue 关系挂到 `#39` 下。
3. 将当前活动链切换为 `WBS 0 -> WBS 1 -> WBS 1.8 -> WBS 1.8.1`。
4. 本轮直接启动 `WBS 1.8.1` 与 `WBS 1.8.2` 的最小实现，不再停留在纸面拆分。
