# WBS 3.1 BaseLab 工具化评论处理

## 评论来源

1. `#4 WBS 3.1 交互画布与渲染基座选型`
   - 评论时间：`2026-04-03T06:20:42Z`
   - 评论编号：`4182103724`
   - 评论要点：
     - 当前 `WBS 3.1` 的对比验证页效果可接受。
     - 希望将该验证页独立出来，不再嵌入主工程根访问路径。
     - 希望建立一个专门的 `BaseLab` 页面，为后续各节点的验证工作提供统一入口。
     - 要求本轮先创建节点、给出验证工具构建方案，不立刻编码。

## 技术判断

### 1. 该意见合理，且已经超出 `WBS 3.1` 单节点边界

1. 当前 `WBS 3.1 Base Lab` 的初衷是服务“交互画布与渲染基座选型”。
2. 用户现在要求它进一步承担“后续多个节点的通用验证入口”角色，这已经从单节点验证页演变为跨节点工具能力。
3. 因此不应继续把后续工具化演进硬塞进 `WBS 3.1`，否则会混淆“本节点选型结论”和“跨节点验证基础设施”。

### 2. 新节点应挂在 `WBS 1`，而不是继续挂在 `WBS 3` 或 `WBS 2`

1. 该工作当前要求是“先拿出方案，不编码”，本质上属于实现前的工具架构设计。
2. 它不是 `WBS 3` 的单一共性机制细节，因为未来知识库、关系图、多轨、观察输出都可能复用它。
3. 它也不应直接并入 `WBS 2`，因为 `WBS 2` 关注核心对象模型与文件化边界，当前轮次并不进入 BaseLab 实施。
4. 结论：将本轮新节点挂到 `WBS 1`，新增为 `WBS 1.8 BaseLab验证工具构建方案`，作为“进入进一步实施前的补充设计节点”。

### 3. BaseLab 工具化是可行的，但不推荐继续复用主工作台根路径

1. 如果继续把验证页挂在主工作台根访问路径下，产品页面与实验页面会共享顶层导航和分析上下文，后续只会越来越重。
2. 用户强调“降低代码分析关联性”和“不污染主工程”，这意味着验证工具应在入口、路由和模块边界上都与正式产品隔离。
3. 结论：BaseLab 应设计成独立入口，而不是继续作为 `apps/web` 根页面中的嵌入式模式。

### 4. 推荐采用“同仓库独立应用”方案，而不是“同应用子页”方案

1. 同应用子页：
   - 成本最低
   - 但会继续与主工作台共享入口、依赖和分析上下文
2. 同仓库独立应用：
   - 仍可复用 monorepo 的构建链、共享包和样例数据
   - 同时能把验证入口、实验路由和实验组件与正式产品明确隔离
3. 仓库外独立实验仓：
   - 隔离最强
   - 但会破坏当前 WBS、文档、样例数据和共享包的同步效率
4. 结论：后续若进入编码，优先采用“同仓库独立应用 `BaseLab`”方案。

## 执行决策

1. 本轮不编码，只做节点与方案设计。
2. 新增 `WBS 1.8 BaseLab验证工具构建方案`。
3. 在本地文档中补齐：
   - `WBS 1.8` 设计说明
   - `WBS 1.8` 研制计划
   - `WBS 1` 父节点与索引文档更新
4. 在 GitHub 上同步：
   - 新建 `WBS 1.8` issue
   - 将其挂入 `WBS 1`
   - 将当前活动链路切为 `WBS 0 -> WBS 1 -> WBS 1.8`
5. `WBS 3.1` 保持现状，不在本轮继续扩大实现范围。

## 不执行项

1. 本轮不把当前 `apps/web` 中的 `WBS 3.1 Base Lab` 迁移为独立应用。
2. 本轮不删除现有验证页实现，避免在方案未确认前破坏已形成的验证证据。
3. 本轮不冻结未来 BaseLab 实施节点编号，待用户确认方案可行后再新增实施节点。
