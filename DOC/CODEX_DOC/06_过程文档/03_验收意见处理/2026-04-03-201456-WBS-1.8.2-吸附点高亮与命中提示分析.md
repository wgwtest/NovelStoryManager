# WBS 1.8.2 吸附点高亮与命中提示分析

## 评论来源

1. GitHub issue：`#41 WBS 1.8.2 WBS 3.1实验迁移与入口解耦`
2. 评论编号：`4183221099`
3. 评论时间：`2026-04-03T12:13:52Z`
4. 评论原意：
   - 当前已经具备吸附点点击画线和连线能力
   - 但鼠标点击或移动到吸附点时，缺少明显提示或高亮，导致难以准确判断
   - 希望了解该功能在三种方案上的可行性、开销和稳定性

## 需求理解

1. 用户不是在否定当前连线方案，而是在指出“端口可发现性不足”。
2. 当前最需要补的是端口的状态提示，而不是继续扩充新交互对象。
3. 用户还要求把该提示能力和三种渲染方案的实现代价、稳定性一并说清楚。

## 代码现实核对

1. `DOM + CSS` 与 `DOM + SVG` 当前已经把端口做成 DOM 按钮，因此具备天然的 hover/focus 入口。
2. `Canvas` 当前只有命中检测和重绘，没有独立 DOM 端口，因此提示必须依赖坐标命中后重绘。
3. 现有连线状态机已经有：
   - 输出端口起拖
   - 动态连线预览
   - 输入端口落点连接
4. 当前缺的是“端口状态层”，至少应区分：
   - 普通悬停
   - 当前连线源端
   - 当前可落点目标

## 三种方案评估

### 1. DOM + CSS

1. 可行性：高
2. 开销：低
3. 稳定性：高
4. 原因：
   - 端口本身就是 DOM 按钮
   - 只需补 class / data-state 与 CSS 视觉反馈
   - 对 `chrome mcp` 和自动化定位最友好

### 2. DOM + SVG

1. 可行性：高
2. 开销：低到中
3. 稳定性：高
4. 原因：
   - 端口仍是 DOM，提示层和 `DOM + CSS` 一样简单
   - 额外只需让 SVG 连线预览与目标高亮同步
   - 这是当前最平衡、最稳定的路线

### 3. Canvas

1. 可行性：中到高
2. 开销：中
3. 稳定性：中
4. 原因：
   - 没有天然 DOM hover，只能在 `pointermove` 时做命中检测
   - 命中后需要重绘高亮环、目标环或源端状态
   - 一旦缩放、平移、命中半径处理不当，就更容易出现“提示抖动”或“误判”

## 本轮设计结论

1. 该意见合理，应在当前 `WBS 3.1` 验证页直接补齐。
2. 推荐采用统一端口状态模型，三种方案共享同一套语义：
   - `idle`
   - `hover`
   - `source`
   - `target`
3. `DOM + CSS` 与 `DOM + SVG`：
   - 用端口按钮的状态属性和 CSS 高亮表达
4. `Canvas`：
   - 用命中检测结果驱动重绘高亮环
5. 本轮不扩展复杂的“无效目标/兼容性校验”矩阵，只先补清晰可见的命中提示，避免范围失控。

## 本轮执行决定

1. 先补失败测试，覆盖：
   - 端口 hover 状态可被识别
   - 连线进行中时，源端与目标端有不同提示状态
2. 再实现三种基座的统一高亮提示。
3. 同轮更新：
   - `WBS 3.1` 设计说明中的可行性、开销、稳定性结论
   - 对应研制计划与验收清单
   - 自测报告、交接文档和 GitHub issue 评论
