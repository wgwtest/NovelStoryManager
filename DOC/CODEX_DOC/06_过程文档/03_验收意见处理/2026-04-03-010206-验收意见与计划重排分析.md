## 问题归类

### 1. 验收清单口径错误

- 用户明确反对人工验收清单里出现“手工读代码确认实现存在”。
- 用户明确反对人工验收清单里出现 `npm test / build / typecheck` 之类开发者自测动作。
- 结论：后续必须严格拆分“用户人工验收清单”和“开发者自测报告”。

### 2. WBS 2.2 规范示范力度不足

- 当前 `WBS 2.2` 只有基础正反样例，缺少一个“小型但字段覆盖尽量全面”的样例数据。
- 结论：在 `WBS 2.2` 成果物中新增高字段覆盖样例数据，并把它纳入验收方法。

### 3. WBS 3/4/5 顶层结构错误

- 当前顶层是“一个大章节 + 一个叶子节点”的薄结构，无法表达真实分解。
- 用户要求先有共性方案，再进入三视图分别的要素设计、能力验证、功能验证和样式验证。
- 结论：重排为 `WBS 3` 三视图共性机制，随后 `WBS 4-6` 分别承载知识库、关系图、多轨视图，并把观察输出层顺延为 `WBS 7`。

### 4. 当前实现已暴露真实产品问题

- `WBS 3.1` 已暴露检查器布局问题和引用跳转语义不清问题。
- `WBS 4.1` 已暴露关系图能力不可观察、联通性判定不可见的问题。
- 结论：这些问题不能再挂在“增强”之类抽象节点下，必须落到新的“能力验证 / 功能验证 / 样式验证”子节点中。

## 处理结论

1. 旧版 `WBS 3-6` 计划文档归档到 `DOC/CODEX_DOC/06_过程文档/02_历史计划/`，保留为第一版尝试记录。
2. 当前主线切换为 `WBS 0 -> WBS 3 -> WBS 3.1`。
3. `WBS 2.2` 进入修订状态，补齐高字段覆盖样例数据与验收口径修正。
4. `WBS 3-7` 使用新树重新承载后续开发。
5. 后续 GitHub issue / Project 应与本次树形重排同步。
