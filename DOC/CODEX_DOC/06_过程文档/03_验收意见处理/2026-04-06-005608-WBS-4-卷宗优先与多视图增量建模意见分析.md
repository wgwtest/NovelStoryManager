## 结构化理解

### 1. 关于 WBS 4 的命名口径

用户的核心意见不是否定当前四个工作面，而是要求上位设计文档不要把 `WBS 4` 锁死成“四视图”。

更准确的理解应为：

1. `WBS 4` 是“多视图设计”父节点
2. 当前只确定了首批四个工作面
3. 后续仍允许继续增加新的工作面
4. 因此设计说明、研制计划和 issue 正文都必须显式写出“当前首批四个”而不是“系统只有四个”

### 2. 关于模型层建设方式

用户明确指出，在项目初期阶段设计一份一次性完备的 `Model` 不现实。

这条评论的真正约束不是“模型先别做”，而是：

1. 允许在视图开发过程中增量补齐迫切需要的模型结构
2. 这种补齐必须保持对象真源与视图状态分层不变
3. 补齐后的模型结构必须回写到数据规范、schema、样例数据和测试中
4. 不允许为了某个视图的局部方便，直接把临时状态写成对象事实

### 3. 关于当前阶段主线

用户明确要求当前或下一阶段任务重点围绕 `卷宗` 推进。

这意味着：

1. `WBS 4.1` 应成为当前优先执行分支
2. `WBS 4` 父节点需要在设计和计划层显式反映这一优先级
3. 后续 `蓝图推演 / 剪辑编排 / 场景调度` 仍保留在树上，但本轮不抢占 `卷宗` 主线资源

### 4. 关于卷宗开发路线

用户对 `卷宗工作面设计` 本身基本认可，但担忧真实数据积累不足，提出新的执行方式：

1. `WBS 4.1.2 / 4.1.3 / 4.1.4` 先在独立页面进行验证开发
2. 该独立页面可以放在 `BaseLab`
3. 当前阶段应先使用模拟数据或样例项目数据完成能力、功能和样式验证
4. 再新增 `WBS 4.1.5`，专门负责把已验证的卷宗工作面融合回主工作台

这说明 `WBS 4.1` 需要从“设计后直接进主界面实现”改为“两阶段路线”：

1. 独立验证阶段
2. 主工作台融合阶段

## 问题归因

1. 当前 `WBS 4` 设计说明虽然已解释 `MVC`，但仍使用了过多“四个工作面”的封闭表述，扩展性口径不够明确。
2. `WBS 2` 与 `WBS 2.2` 文档强调了真源分层，但还没有把“视图推进中增量补模”的策略正式写进规范。
3. `WBS 4.1` 目前只拆到 `4.1.4`，缺少用户要求的“主工作台融合”节点。
4. `WBS 4.1.2 / 4.1.3 / 4.1.4` 现有成果物仍以 `apps/web` 为主，和用户提出的 `BaseLab + 模拟数据` 路线不一致。

## 本轮处理策略

1. 将 `WBS 4` 的设计说明与研制计划改写为“多视图设计，当前首批四个工作面”的口径，并补入扩展性约束。
2. 在 `WBS 4` 顶层设计中加入“模型增量补齐策略”，明确视图开发中允许增量补模，但必须同步回写规范与测试。
3. 在 `WBS 2` 与 `WBS 2.2` 文档中补入“增量扩展但分层不变”的约束，避免后续误解为模型已经一次性冻结。
4. 将 `WBS 4.1` 改为卷宗优先路线，并明确 `4.1.2 / 4.1.3 / 4.1.4` 先在 `BaseLab` 用模拟数据验证。
5. 新增 `WBS 4.1.5 卷宗主工作台融合` 节点，用于承接 `BaseLab` 验证后的正式接入。
6. 同一轮同步更新 GitHub issue 正文、子节点结构和 Project 投影视图，避免再次出现本地已改但远端仍沿用旧口径。

## 预期结果

1. `WBS 4` 的上位设计将从“当前四个页面”升级为“可持续扩展的多视图体系”。
2. `WBS 2` 与 `WBS 4` 将形成一致口径：对象真源和视图状态分层不变，但模型允许按视图开发节奏增量补齐。
3. `WBS 4.1` 将明确转为当前主线，并采用 `BaseLab` 先验、主工作台后融的两阶段实施方式。
4. 本地文档、GitHub issue 和 GitHub Project 的树结构与执行顺序将重新一致。
