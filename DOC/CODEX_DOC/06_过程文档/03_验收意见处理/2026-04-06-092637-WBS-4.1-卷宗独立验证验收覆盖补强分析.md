## 结构化理解

### 1. 用户并未否定当前实现

用户已经在本地验收清单上逐项勾选通过，说明当前 `BaseLab` 卷宗独立验证页的首轮实现已经满足最小能力链。

因此本轮问题不应理解为：

1. 卷宗页功能失败
2. 样式不可用
3. 交互链断裂

而应理解为：

1. 当前人工验收脚本只覆盖了最小主链
2. 还没有把现有页面中已经具备的更多能力纳入验收
3. 验收脚本的密度不足以支撑后续 `WBS 4.1.5` 融合判断

### 2. “内容缺很多”更接近验收覆盖不足，而不是立即要求大改功能

结合现有实现，卷宗独立验证页实际上已经包含但尚未充分验收的内容至少还有：

1. 左侧九类对象分类切换
2. 关键词筛选行为
3. 引用回查链路
4. 审校队列点击跳转
5. 检查器的 `Core Fields / Additional Fields / 只读字段` 分组
6. 字段改写后正文与状态区同步

也就是说，问题主要在于：

1. 验收清单没有覆盖完整观察面
2. 自动化测试也只锁住了最核心的四条链

### 3. 当前最合理的处理方式是“补强验证”，而不是立刻扩写主功能

在没有更具体产品意见前，直接新增大量卷宗功能会产生两个问题：

1. 容易偏离当前 `WBS 4.1.2 / 4.1.3 / 4.1.4` 的独立验证边界
2. 会把“验收脚本不够完整”误处理成“产品需求发生变化”

因此本轮应先做：

1. 扩充自动化测试覆盖
2. 扩充人工验收清单覆盖
3. 扩充自测报告中的关键验证结论

而不是直接把 `WBS 4.1.5` 的融合工作提前混入当前节点。

## 问题归因

1. 首轮验收清单过于聚焦最短主链，未覆盖回查链、审校链和检查器分组链。
2. 首轮自动化测试覆盖不足，无法为“内容更完整”的人工验收提供足够背书。
3. `19 / 20 / 21` 三份节点计划虽然已进入 `待人工验收`，但验收覆盖描述仍偏薄。

## 本轮处理策略

1. 先新增本分析文档，明确本轮反馈的真实含义是“验收覆盖补强”。
2. 在 `apps/base-lab/src/components/DossierAbilityLab.test.tsx` 中补充：
   - 关键词筛选
   - 引用回查跳转
   - 审校队列跳转
   - 检查器分组可见
3. 必要时补充最小 UI 文案或可达性标识，让这些行为可以稳定测试和人工观察。
4. 扩充本地验收清单，将原有已通过条目保留，并追加补充验收项。
5. 更新自测报告和相关节点文档中的验收覆盖描述。

## 预期结果

1. 当前卷宗独立验证页的通过结论将不再只依赖最短主链。
2. `WBS 4.1.2 / 4.1.3 / 4.1.4` 的人工验收脚本将更接近后续融合决策所需的观察密度。
3. 在没有新增产品需求的前提下，本轮修改仍保持在“独立验证补强”边界内。
