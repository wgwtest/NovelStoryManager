# GitHub 评论优化分析

## 评论来源

1. `#10 WBS 2 核心对象模型与文件化项目`
   - 评论：题目中的“项目”容易产生歧义，希望移除。
2. `#4 WBS 3.1 交互画布与渲染基座选型`
   - 评论：如果主交互层使用 `Canvas`，是否会影响 `chrome mcp` 与自动化测试。
   - 评论：关系图节点希望更接近 Unreal 蓝图节点效果，而不是低信息量的普通卡片与连线。
3. `#9 WBS 1 需求与方案`
   - 评论：需要新增“高保真（至少中保真）原型设计”节点，并提供能自动生成高保真原型界面的方案。

## 技术判断

### 1. `WBS 2` 标题去掉“项目”是合理意见

1. 当前节点关注的是对象模型、文件化存储边界和导入导出约束。
2. “文件化项目”容易被读成“项目管理本身”，不如“文件化”直接。
3. 结论：将当前有效标题统一调整为 `WBS 2 核心对象模型与文件化`，不改 `WBS 2.1 项目底座实施` 这个已稳定叶子节点。

### 2. `Canvas` 作为主交互载体不适合本项目当前阶段

1. 本项目当前的浏览器自动化与 `chrome mcp` 工作流，本质上依赖 DOM 与可访问性树快照。
2. 纯 `Canvas` 会把节点、端口、边锚点和工具按钮压扁成位图区域，自动化只能退化成坐标点击，回归测试会更脆弱。
3. 关系图与多轨视图都需要真实的按钮、端口、属性入口、尺寸输入和拖拽热点，DOM/SVG 更容易维持语义、可测性与可编辑性。
4. 结论：保留 `DOM 节点/块 + SVG 连线 + 共享视口状态` 的主方案；`Canvas` 只允许承担网格、背景、热区预览等非关键装饰层。

### 3. Unreal 蓝图节点风格属于有效约束

1. 这不是单纯审美意见，而是信息密度与交互可见性的要求。
2. “标题条 + 左右端口列 + 类型标签 + 吸附锚点 + 连线高亮 + 有效/无效连接反馈”会直接影响后续能力验证是否足够清楚。
3. 该约束应写进 `WBS 3.1` 的基座选型与 `WBS 5.1` 的关系图要素设计，而不是只留在聊天里。

### 4. 新增中高保真原型节点是合理意见

1. 当前 `WBS 1` 已完成低保真原型、交互验证、技术选型与任务拆解，但用户新增了“需要中高保真原型基线”的要求。
2. 直接回改已验收的 `WBS 1.5`、`WBS 1.6` 编号会破坏已经稳定的验收与引用关系。
3. 结论：新增 `WBS 1.7 中高保真原型设计与生成方案`，作为 `WBS 1` 的补充节点。
4. 为了不重写整段既有文件前缀，`03_研制计划/` 采用字母后缀插入位：`08A-WBS-1.7-...-研制计划.md`。
