## 结构化理解

### 1. 关于 WBS 4 父节点

用户的核心意见不是“继续补几段描述”，而是要求补上 `WBS 4` 的总设计说明，明确回答以下问题：

1. 一套数据与多个视图之间到底是什么关系
2. 是否严格遵循 `MVC`
3. 视图之间是否存在直接耦合
4. 视图内部子区域之间如何协作
5. 哪些东西是共享的，哪些东西必须保持独立可替换

这说明当前 `WBS 4` 的缺口不是叶子节点描述不够长，而是缺少一个上位设计约束文档。

### 2. 关于四个工作面父节点

用户认为当前 `卷宗 / 蓝图推演 / 剪辑编排 / 场景调度` 的设计说明过于停留在“要素列表”，没有回答：

1. 该工作面存在的目的
2. 该命名背后的产品语义
3. 它与其他工作面的职责边界
4. 它由哪些数据驱动
5. 它要求模型层提供哪些数据
6. 它内部各区域如何工作

这说明每个工作面都需要一份父级设计说明，不能只保留 `WBS 4.x.1` 的要素设计文档。

### 3. 关于 MVC 约束

用户明确要求：

1. 四个视图在视图实现层严格解耦
2. 四个视图都由同一套项目数据驱动
3. 允许共享基础绘制理念、对象选择语义和公共外壳
4. 不允许一个视图直接依赖另一个视图的内部实现
5. 各视图理论上应可以独立替换

这意味着本轮设计必须把“共享 Shell / 共享 Controller 契约 / 共享 Model”和“独立 View 实现”同时写清楚。

### 4. 关于 WBS 3.1

用户已明确两条结论：

1. 后续实现依据选择 `DOM + SVG`
2. `WBS 3.1` 可以标记完成

因此 `WBS 3.1` 不应继续保留“待人工验收”状态，而应在本地文档、issue 契约和 Project 中同步收口为已验收。

## 问题归因

1. 之前的工作更多围绕原型重排和任务树重排，导致 `WBS 4` 的上位设计文档缺位。
2. `WBS 4.1.1` 到 `WBS 4.4.1` 的要素设计文档被误当成父级设计文档使用，语义层级错位。
3. `WBS 3.1` 的结论虽然已被用户确认，但状态同步没有在同轮闭环。
4. `WBS 4` 相关 issue 成果物路径与本地实际文档名已开始出现漂移，需要一并收敛。

## 本轮处理策略

1. 新增 `WBS 4` 顶层设计说明，专门解释多视图共享模型、`MVC` 约束和独立替换原则。
2. 新增 `WBS 4.1` 到 `WBS 4.4` 四份父级工作面设计说明，回答“为什么存在、靠什么数据驱动、与其他工作面如何分工”。
3. 保留并修正 `WBS 4.x.1` 要素设计文档，让其回到“页面构成与元素约束”的层级。
4. 同步更新 `WBS 4` 与 `WBS 4.x` 的研制计划成果物和验收方法，使之引用新的父级设计说明。
5. 将 `WBS 3.1` 本地文档与远端跟踪状态同步为已验收，并把“`DOM + SVG` 为后续实现基线”写成冻结结论。
6. 在同一轮同步 GitHub issue 正文与 Project `Status`，避免再次出现“文档已改、远端未改”的漂移。

## 预期结果

1. `WBS 4` 将具备可以直接指导实现的总设计约束。
2. 四个工作面将分别具备父级设计说明和子级要素设计说明，不再混层。
3. `WBS 3.1` 的结论与状态将正式收口。
4. 本地文档、GitHub issue、GitHub Project 的口径重新一致。
