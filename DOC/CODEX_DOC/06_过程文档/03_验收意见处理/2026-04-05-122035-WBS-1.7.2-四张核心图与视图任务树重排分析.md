# WBS 1.7.2 四张核心图与视图任务树重排分析

## 输入意见

用户在 `#43` 的最新评论与当前会话中进一步澄清了两点：

1. `WBS 1.7.2` 不再保留 `8` 套路线，也不再保留“每套 `5` 张图”的表达方式。
2. 当前只保留 `4` 张核心图，分别围绕 `卷宗`、`蓝图推演`、`剪辑编排`、`场景调度` 四个角度继续深化。
3. 这次变化不仅是原型深化角度变化，产品页面结构本身也要随之调整。
4. 当前 GitHub 任务树不够合理：
   - `WBS 3` 应从“三视图共性机制”调整为“多视图共性机制”
   - 当前 `WBS 4 / 5 / 6` 应提升归类到一个更大的“视图设计”父节点下

## 意见合理性分析

这组意见合理，原因如下：

1. 当前原型阶段的真正目标已经从“多方案海选”收束为“确认主工作面方向”，继续保留 `8` 套路线只会增加理解成本。
2. 用户已经明确判断哪些路线无价值，因此应当直接删除，而不是继续维持对称结构。
3. “卷宗 / 蓝图推演 / 剪辑编排 / 场景调度”比“知识库 / 关系图 / 轨道”更接近当前被验证过的工作面心智，因此应当允许它们上升为页面结构主轴。
4. 当前 `WBS 4 / 5 / 6` 作为并列一级任务虽然能看懂，但从树结构上看，确实缺少一个“视图设计”父节点来承载同类任务。
5. `WBS 3` 的职责本来就不是只服务三页，而是服务所有视图共享的坐标、元素、编辑协议与验证基线，因此改为“多视图共性机制”更准确。

## 本轮设计收束结论

### WBS 1.7.2

从当前开始，`WBS 1.7.2` 的成果物从“8 套路线比较”改为“4 张核心图深化”：

1. `01-卷宗工作面`
2. `02-蓝图推演工作面`
3. `03-剪辑编排工作面`
4. `04-场景调度工作面`

每个方向只保留 `1` 张核心图，但要求：

1. 图中要把当前工作面的核心结构讲清楚
2. 不能继续堆叠用户当前无法一眼理解的术语块
3. 产品主页面结构应改为围绕这四个工作面组织

### 任务树

任务树按以下方向重排：

1. `WBS 3` 重命名为 `多视图共性机制`
2. `WBS 4` 调整为新的父节点 `视图设计`
3. 原 `知识库工作台 / 关系图工作台 / 多轨编排视图` 下沉为 `WBS 4` 的子节点
4. 这些子节点下再继续保留“要素设计 / 能力验证 / 功能验证 / 样式验证”的叶子结构

## 实施策略

1. 先更新本地设计说明、研制计划、执行说明和过程文档
2. 再把 `scripts/generate-wbs-1.7.2-prototypes.*` 的测试契约改成 `4` 张核心图
3. 再重写 `WBS 1.7.2` 原型生成与导出结果
4. 最后同步 GitHub `#43` 的 issue 契约与评论，并重排 `WBS 3 / 4 / 5 / 6` 的 issue 树

## 风险与约束

1. `WBS 4 / 5 / 6` 重排后，现有本地文档和 GitHub issue 标题编号会一起变化，必须整轮同步。
2. 已经完成的旧 `WBS 1.7.2` 八套原型目录会被新的四张核心图文件替换。
3. 任务树调整优先保证父子关系与语义正确，其次才是编号整齐。

## 结论

本轮应把 `WBS 1.7.2` 从“路线比较”切换到“核心工作面深化”，并把视图相关 WBS 树同步收束为“多视图共性机制 -> 视图设计 -> 各具体视图”的结构。这样更符合当前用户已经给出的明确取舍。
