# 本地工程策略映射

## 目标

在不改变正常研发节奏的前提下，将 NovelStoryManager 的文档、计划、验收和交接统一收敛到项目级工程策略结构中，避免后续多轮开发时出现文档散落、计划命名漂移和验收证据断链。

## 文档根决策

1. 项目初始化前不存在 `DOC/CodexAnylyse/`
2. 项目初始化前不存在 `DOC/CODEX_DOC/`
3. 本轮创建并采用 `DOC/CODEX_DOC/` 作为项目文档根

选择理由：

1. 该目录符合既定工程策略要求
2. 与仓库根目录解耦，便于长期扩展
3. 适合作为设计、计划、验收、自测、交接的统一归档入口

## 本项目采用的协作策略

1. 先模型与契约，再实现与页面
2. 先需求边界、信息架构和原型，再冻结技术方案并写详细实施计划
3. 通过依赖关系和时间规划推进，不单独维护里程碑轴
4. 用户未验收前，不把阶段标记为完成
5. 允许推进低耦合 Prep，不把依赖上游稳定输出的 Core 误判为正式开始
6. 对强交互编辑器，交互技术栈必须依托原型目标和高风险交互验证结果，不允许脱离界面目标先拍板

## 三层协同映射

### 1. Tracker 层

当前已建立 GitHub Project v2：

1. Project：`NovelStoryManager Delivery Roadmap`
2. URL：<https://github.com/users/wgwtest/projects/3>

当前处理策略：

1. GitHub Project 负责 WBS 树、状态、时间规划和路线可视化
2. 本地文档继续负责长设计、技术选型、详细实施计划、自测和交接

Project 计划设计原则：

1. Project 的首要职责是表达任务树、依赖关系和当前主线，不是堆放任务标题
2. 用户阅读 Project 时，应能直接看出“先做什么、后做什么、当前做到哪条链路”
3. 不能帮助理解结构、依赖或时间安排的信息，不应作为 Project 的常驻管理轴

树形表达规则：

1. 只要任务之间存在稳定父子分解关系，Project 就必须使用父 issue / 子 issue 表达
2. `WBS 0` 作为路线根节点，`WBS 1` 到 `WBS 6` 作为一级模块父节点，执行任务挂到对应叶子节点
3. 只有确实不存在父子归属关系的事项，才允许作为同层并列条目存在
4. 不允许仅靠标题前缀或字段值模拟层级，却在结构上保持扁平
5. 父节点和叶子节点名称都应概括、易懂、可直接理解，不使用“项目启动”“架构基线”这类信息量不足的抽象词

字段使用规则：

1. `Status` 用于表达任务当前执行状态
2. `Work Type` 用于表达 `Prep / Core / Docs`
3. `Start Date` 与 `Target Date` 用于表达时间安排
4. 树关系优先通过原生父 issue / 子 issue 表达
5. `Depends On` 写入 issue 契约正文，不依赖聊天上下文补全理解

里程碑轴规则：

1. 当前项目默认不使用 `M1 / M2 / M3 / M4` 作为 Project 的常驻管理轴
2. 里程碑节点只有在用户明确要求，或存在对外发布/阶段验收需要时才可重新引入
3. 即使未来重新启用里程碑，WBS 树仍然是第一导航轴，里程碑只能作为辅助观察维度
4. 不允许在 WBS 树未清晰前，先用里程碑字段替代任务分解

推进表达规则：

1. 主导航轴只使用 `WBS`
2. 任务先后顺序通过 `Depends On` 表达
3. 时间安排通过 `Start Date` 和 `Target Date` 表达
4. 不再使用 `M1 / M2 / M3 / M4` 作为 Project 的常驻管理轴
5. 当前主线必须能够被明确描述为一条从根节点到叶子节点的链路

### 2. Execution Contract 层

执行契约将统一进入 `DOC/CODEX_DOC/契约草案/`

最小契约字段固定为：

1. `工作目标`
2. `工作内容`
3. `潜在难点`
4. `Depends On`
5. `成果物`
6. `验收方法`
7. `当前状态`
8. `上级节点`

字段编写要求：

1. `工作目标` 必须写清楚这项工作完成后世界上新增了什么，不写口号式描述
2. `工作内容` 必须列出具体动作、具体模块或具体文档，不写泛泛表述
3. `潜在难点` 必须写真实风险或实现难点，不能为了凑字段写空话
4. `成果物` 必须写成可点名检查的东西，例如某份文档、某个代码模块、某个 Project 结构调整
5. `验收方法` 必须写成可执行的核验动作，而不是抽象判断
6. 当同一 `WBS` 同时存在“节点说明页”和“正文文档”时，`成果物` 默认优先写正文文档，节点说明页只作为导航文档引用，除非该任务本身就是修改节点说明结构

验收方法规则：

1. 文档类任务：
   - 直接写人工核验方法
   - 必须指出核验哪个文件，以及检查哪些条目是否存在
2. Project / issue 结构类任务：
   - 直接写人工核验 GitHub Project、issue 树、字段值或日期的方法
   - 必须指出检查哪些 issue、哪些字段、哪些父子关系
3. 代码类任务：
   - 必须提供测试命令、启动命令或手工验证步骤
   - 必须写清预期结果，例如“命令退出码为 0”“页面出现某元素”“接口返回某结果”

禁用写法：

1. 不再使用 `验收入口`
2. 不再使用空泛的 `验收标准`
3. 不允许使用“关系明确”“体验提升”“功能可用”这类无法追踪的句子作为主要验收描述
4. 不允许让 issue 依赖聊天上下文才能理解如何验收

### 3. Local Docs 层

项目长文档统一进入 `DOC/CODEX_DOC/`，按下列类别归档：

1. 稳定说明文档：文档根
2. 设计稿：`设计稿/`
3. 契约草案：`契约草案/`
4. 开发计划：`开发计划/`
5. 自测报告：`自测报告/`
6. 会话交接：`会话交接/`
7. 验收类文档：`验收清单/`、`验收记录/`、`验收结论/`

文档职责分工规则：

1. 文档根下的 `WBS` 文档默认承担节点说明、阅读顺序、状态索引和上下游衔接职责
2. `设计稿/` 下的文档承担需求正文、方案正文、模型正文等稳定内容职责
3. 同一 `WBS` 如果同时存在节点说明页和正文文档，两者标题必须能直接区分，不能再使用同名标题造成重复感
4. 节点说明页必须链接到正文入口，但不重复展开整份正文内容
5. 即使暂时没有单独正文，`WBS` 根文档也建议显式追加 `节点说明`，提前固定导航文档语义

## 命名规则

1. 稳定计划文档按 WBS 节点命名
2. `WBS` 节点说明页允许在节点名后追加 `节点说明`
3. 设计稿按日期加主题命名
4. 验收类文档按 `YYYY-MM-DD-HHMMSS-文件名.md` 命名
5. 历史计划进入 `开发计划/历史计划/`
