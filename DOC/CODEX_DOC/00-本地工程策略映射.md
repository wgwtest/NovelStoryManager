# 本地工程策略映射

## 目标

在不改变正常研发节奏的前提下，将 NovelStoryManager 的文档、计划、验收和交接统一收敛到项目级工程策略结构中，避免后续多轮开发时出现文档散落、计划命名漂移和验收证据断链。

## 文档根决策

1. 本项目当前不存在 `DOC/CodexAnylyse/`
2. 本项目当前不存在 `DOC/CODEX_DOC/`
3. 本轮创建并采用 `DOC/CODEX_DOC/` 作为项目文档根

选择理由：

1. 该目录符合既定工程策略要求
2. 与仓库根目录解耦，便于长期扩展
3. 适合作为设计、计划、验收、自测、交接的统一归档入口

## 本项目采用的协作策略

1. 先模型与契约，再实现与页面
2. 先需求边界和技术方案，再写详细实施计划
3. 通过依赖关系和时间规划推进，不单独维护里程碑轴
4. 用户未验收前，不把阶段标记为完成
5. 允许推进低耦合 Prep，不把依赖上游稳定输出的 Core 误判为正式开始

## 三层协同映射

### 1. Tracker 层

当前已建立 GitHub Project v2：

1. Project：`NovelStoryManager Delivery Roadmap`
2. URL：<https://github.com/users/wgwtest/projects/3>

当前处理策略：

1. GitHub Project 负责 WBS 树、状态、时间规划和路线可视化
2. 本地文档继续负责长设计、技术选型、详细实施计划、自测和交接

树形表达规则：

1. 只要任务之间存在稳定父子分解关系，Project 就必须使用父 issue / 子 issue 表达
2. `WBS 0` 作为路线根节点，`WBS 1` 到 `WBS 6` 作为一级模块父节点，执行任务挂到对应叶子节点
3. 只有确实不存在父子归属关系的事项，才允许作为同层并列条目存在
4. 不允许仅靠标题前缀或字段值模拟层级，却在结构上保持扁平

推进表达规则：

1. 主导航轴只使用 `WBS`
2. 任务先后顺序通过 `Depends On` 表达
3. 时间安排通过 `Start Date` 和 `Target Date` 表达
4. 不再使用 `M1 / M2 / M3 / M4` 作为 Project 的常驻管理轴

### 2. Execution Contract 层

执行契约将统一进入 `DOC/CODEX_DOC/契约草案/`

最小契约字段固定为：

1. `目标`
2. `Owner`
3. `Write Scope`
4. `Depends On`
5. `验收入口`
6. `验收标准`
7. `输出物`
8. `证据链接`
9. `当前状态`
10. `上级节点`

### 3. Local Docs 层

项目长文档统一进入 `DOC/CODEX_DOC/`，按下列类别归档：

1. 稳定说明文档：文档根
2. 设计稿：`设计稿/`
3. 契约草案：`契约草案/`
4. 开发计划：`开发计划/`
5. 自测报告：`自测报告/`
6. 会话交接：`会话交接/`
7. 验收类文档：`验收清单/`、`验收记录/`、`验收结论/`

## 命名规则

1. 稳定计划文档按 WBS 节点命名
2. 设计稿按日期加主题命名
3. 验收类文档按 `YYYY-MM-DD-HHMMSS-文件名.md` 命名
4. 历史计划进入 `开发计划/历史计划/`
