# 本地工程策略映射

## 目标

在不改变正常研发节奏的前提下，将 NovelStoryManager 的需求、设计、计划、研发、测试和过程文档统一收敛到项目级工程策略结构中，避免后续多轮开发时出现文档散落、计划命名漂移、验收证据断链和正文入口难以检索的问题。

## 全局继承关系

1. 本项目当前采用的 `CODEX_DOC` 六大类目录结构，已上升为共享工程策略基线。
2. 对应的全局策略入口位于：
   - `/home/wgw/CodexProject/ENGINEERING_WORK_STRATEGY.md`
   - `/home/wgw/.codex/skills/project-engineering-strategy/references/ENGINEERING_WORK_STRATEGY.md`
   - `/home/wgw/.codex/skills/project-engineering-strategy/SKILL.md`
   - `/home/wgw/.codex/skills/project-engineering-strategy/references/PROJECT_STARTUP_TEMPLATE.md`
3. 后续其他项目如需新建或重构本地文档体系，应优先对齐这套分类、命名和归档规则。
4. 已存在历史文档根的项目，可暂时保留根目录名称，但应逐步向相同的内部分类结构靠拢。

## 文档根决策

1. 项目初始化前不存在 `DOC/CodexAnylyse/`
2. 项目初始化前不存在 `DOC/CODEX_DOC/`
3. 本项目采用 `DOC/CODEX_DOC/` 作为统一文档根
4. 文档根只常驻 `README.md` 与 `00-本地工程策略映射.md` 两份核心索引文档

选择理由：

1. 该目录符合既定工程策略要求
2. 与仓库根目录解耦，便于长期扩展
3. 便于以带顺序前缀的分类目录方式承载需求、设计、计划、研发、测试和过程文档
4. 适合后续持续导入新的规范正文、测试证据和交接记录

## 本项目采用的协作策略

1. 先模型与契约，再实现与页面
2. 先需求边界、信息架构和原型，再冻结技术方案并形成单节点计划文档
3. 通过依赖关系和时间规划推进，不单独维护里程碑轴
4. 用户未验收前，不把阶段标记为完成
5. 允许推进低耦合 Prep，不把依赖上游稳定输出的 Core 误判为正式开始
6. 对强交互编辑器，交互技术栈必须依托原型目标和高风险交互验证结果，不允许脱离界面目标先拍板
7. 用户在聊天或文稿评阅中明确给出“通过 / 已确认 / 验收通过”后，必须在同一轮同步更新 GitHub Project、issue 契约和本地状态文档
8. 默认直接在 `main` 分支提交与推送；只有用户明确要求隔离分支，或当前改动需要高风险隔离试验时，才允许新建功能分支
9. 本地文档目录、文件名或分类结构一旦调整，必须在同一轮同步更新 GitHub Issue、GitHub Project 等协作面中的文件路径引用，不允许继续保留旧目录名或 `/home/...` 绝对路径
10. 收到验收批注意见后，先写 `06_过程文档/03_验收意见处理/` 处理文档，再决定改计划、改验收还是改代码
11. 用户人工验收清单只允许出现用户可观察动作和文档核验动作，不允许出现读代码项或 `npm` 命令项
12. 命令执行、单元测试、构建结果统一写入 `05_测试文档/01_自测报告/`
13. 强交互多视图必须先经过“共性机制”节点，再进入各自的“要素设计 / 能力验证 / 功能验证 / 样式验证”

## 三层协同映射

### 1. Tracker 层

当前已建立 GitHub Project v2：

1. Project：`NovelStoryManager Delivery Roadmap`
2. URL：<https://github.com/users/wgwtest/projects/3>

当前处理策略：

1. GitHub Project 负责 WBS 树、状态、时间规划和路线可视化
2. 本地文档负责长设计、单节点计划文档、研发规范、自测和交接

Project 计划设计原则：

1. Project 的首要职责是表达任务树、依赖关系和当前主线，不是堆放任务标题
2. 用户阅读 Project 时，应能直接看出“先做什么、后做什么、当前做到哪条链路”
3. 不能帮助理解结构、依赖或时间安排的信息，不应作为 Project 的常驻管理轴

树形表达规则：

1. 只要任务之间存在稳定父子分解关系，Project 就必须使用父 issue / 子 issue 表达
2. `WBS 0` 作为路线根节点，`WBS 1` 到 `WBS 7` 作为一级模块父节点，执行任务挂到对应叶子节点
3. `WBS 3` 固定作为多视图共性机制父节点，`WBS 4` 固定作为视图设计父节点，再下挂 `卷宗 / 蓝图推演 / 剪辑编排 / 场景调度` 四个执行分支，`WBS 7` 承载观察输出层
4. 只有确实不存在父子归属关系的事项，才允许作为同层并列条目存在
5. 不允许仅靠标题前缀或字段值模拟层级，却在结构上保持扁平
6. 父节点和叶子节点名称都应概括、易懂、可直接理解，不使用“项目启动”“架构基线”这类信息量不足的抽象词
7. 对强交互工作台，不允许继续使用“一个大章节 + 一个叶子节点”的薄结构，必须至少拆成“要素设计 / 能力验证 / 功能验证 / 样式验证”

字段使用规则：

1. `Status` 用于表达任务当前执行状态
2. `Work Type` 用于表达 `Prep / Core / Docs`
3. `Start Date` 与 `Target Date` 用于表达时间安排
4. 树关系优先通过原生父 issue / 子 issue 表达
5. `Depends On` 写入 issue 契约正文，不依赖聊天上下文补全理解

验收通过后的状态同步规则：

1. 用户一旦明确给出“通过 / 已确认 / 验收通过”，必须在同一轮同步更新 GitHub Project 对应节点的 `Status`
2. 当前 Project 的粗粒度完成态默认映射为 `Done`
3. 同一轮还必须同步更新对应 issue 契约中的 `当前状态`
4. 如果本地节点文档、README 或索引文档中暴露了旧状态，也必须同步改掉
5. 不允许出现“设计文稿评阅已写 `【通过】`，但 GitHub Project 仍是 `Todo / In Progress`”的漂移

里程碑轴规则：

1. 当前项目默认不使用 `M1 / M2 / M3 / M4` 作为 Project 的常驻管理轴
2. 里程碑节点只有在用户明确要求，或存在对外发布/阶段验收需要时才可重新引入
3. 即使未来重新启用里程碑，WBS 树仍然是第一导航轴，里程碑只能作为辅助观察维度
4. 不允许在 WBS 树未清晰前，先用里程碑字段替代任务分解

推进表达规则：

1. 主导航轴只使用 `WBS`
2. 任务先后顺序通过 `Depends On` 表达
3. 时间安排通过 `Start Date` 和 `Target Date` 表达
4. 不再使用 `M1 / M2 / M3 / M4` 作为 Project 的常驻管理轴
5. 当前主线必须能够被明确描述为一条从根节点到叶子节点的链路

### 2. Execution Contract 层

执行契约以 GitHub Issue 为主，本地 `DOC/CODEX_DOC/03_研制计划/` 中的单节点文档作为增强版镜像。

最小契约字段固定为：

1. `工作目标`
2. `工作内容`
3. `潜在难点`
4. `Depends On`
5. `成果物`
6. `验收方法`
7. `当前状态`
8. `上级节点`

字段编写要求：

1. `工作目标` 必须写清楚这项工作完成后世界上新增了什么，不写口号式描述
2. `工作内容` 必须列出具体动作、具体模块或具体文档，不写泛泛表述
3. `潜在难点` 必须写真实风险或实现难点，不能为了凑字段写空话
4. `WBS` 叶子节点名称必须直接体现业务对象、规范对象或实现对象，不用“增强”“优化”“契约”这类抽象词替代真正目标，除非任务本身就是改写契约文本
5. `成果物` 必须优先写成仓库内可直接定位的明确路径；只有路径暂时无法预判时，才退回到目录或模块级表达
6. 数据规范、文件规范、导入导出类任务的 `成果物`，至少要同时点名：规范正文、实现文件、测试夹具
7. 数据规范、文件规范、导入导出类任务的 `成果物`，还必须点名 1 个小型但字段覆盖尽量全面的样例数据文件
8. `验收方法` 必须写成可执行的核验动作，而不是抽象判断
9. 当同一 `WBS` 同时存在“节点文档”和“正文文档”时，`成果物` 默认优先写正文文档，节点文档负责承载前置条件、后续节点、实施方式和验收方法
10. GitHub Issue、GitHub Project 和其他共享协作面中的文件路径，统一使用仓库相对路径；如需点击打开，再补充仓库 URL，不使用本机绝对路径

验收方法规则：

1. 文档类任务：
   - 直接写人工核验方法
   - 必须指出核验哪个文件，以及检查哪些条目是否存在
2. Project / issue 结构类任务：
   - 直接写人工核验 GitHub Project、issue 树、字段值或日期的方法
   - 必须指出检查哪些 issue、哪些字段、哪些父子关系
3. 代码类任务：
   - 必须提供测试命令、启动命令或手工验证步骤
   - 必须写清预期结果，例如“命令退出码为 0”“页面出现某元素”“接口返回某结果”
   - 这些命令属于执行契约和自测报告，不应照搬到用户人工验收清单里
4. 数据规范、文件规范、导入导出类任务：
   - 必须同时给出正样例、反样例和往返一致性测试
   - 必须给出高字段覆盖样例数据的核验方式
   - 必须写清样例文件路径、测试命令和预期失败或成功结果
   - 如涉及版本字段，还必须给出版本不兼容场景的失败测试
5. 用户若约定了固定评阅标记方式，本项目默认承认 `【】` 内的明确通过结论属于正式验收信号

禁用写法：

1. 不再使用 `验收入口`
2. 不再使用空泛的 `验收标准`
3. 不允许使用“关系明确”“体验提升”“功能可用”这类无法追踪的句子作为主要验收描述
4. 不允许让 issue 依赖聊天上下文才能理解如何验收

目录职责分工规则：

1. `01_需求分析/`：

2. `02_设计说明/`：
   - 平铺存放总体设计、信息架构、低保真原型、高风险交互验证、技术方案
   - 不为仅含单个文件的设计主题额外建立子目录

   - 平铺存放 `WBS` 节点文档
   - 每个节点只保留一份主文档
   - 一份文档同时覆盖前置条件、后续节点、工作目标、工作内容、实施方式、成果物、时间安排和验收方法
4. `04_研发文档/`：
   - 存放数据结构设计稿、关键业务流程文档、导入导出规范和原型稿等研发正文
5. `05_测试文档/`：
   - 存放自测报告、验收清单、验收记录、验收结论
6. `06_过程文档/`：
   - 存放会话交接、历史计划、验收意见处理及其他过程性归档

文档职责补充规则：

1. `03_研制计划/` 不再拆分为“节点说明 / 执行契约 / 实施计划”三套并行目录
2. 单个 `WBS` 节点只保留一份主文档，避免理解成本随着文档数量上涨
3. 旧版细化实施计划如需保留，统一归档到 `DOC/CODEX_DOC/06_过程文档/02_历史计划/`
4. 对规范本身就是主要交付物的任务，正文优先写入 `04_研发文档/`
5. 核心配置文档 `00-本地工程策略映射.md` 保持在文档根，不随专题迁移

## 命名规则

1. 稳定计划文档按 WBS 节点命名
2. `WBS` 叶子节点标题优先使用直白的业务名词，例如 `数据规范`、`文件规范`、`导入导出规范`、`项目底座实施`
3. `03_研制计划/` 下的节点文档统一按 `NN-WBS-节点编号-主题-研制计划.md` 命名
4. `02_设计说明/` 下的稳定设计正文统一按 `WBS-节点编号-主题.md` 命名，不在文件名前保留日期
5. 无法直接归属具体节点的总设计稿，也应优先挂到最贴近的 `WBS` 编号下，避免同层混入日期命名
6. `04_研发文档/` 正文可继续按日期加主题命名
7. 验收类文档按 `YYYY-MM-DD-HHMMSS-文件名.md` 命名
8. 历史计划进入 `DOC/CODEX_DOC/06_过程文档/02_历史计划/`
9. 一级目录与二级目录都使用固定数字前缀，后续新增同层目录时必须补入顺序号
10. 文件名保留业务语义，不因目录序号策略而追加重复编号
11. `04_研发文档/03_原型图/` 作为静态原型稿、设计导出图和原型说明的固定落盘目录
