## 工作目标

- 选定多视图分支的交互画布与渲染基座，回答“是否统一使用同一种画布技术、哪些视图允许例外、拖拽和缩放靠什么实现”。
- 把选型结论落成可直接观察的 `BaseLab` 独立实验入口，让用户基于真实页面效果确认而不是只看文字分析。

## 工作内容

- 对比 DOM、SVG、Canvas 及现有交互库的适配性。
- 区分知识库、关系图、多轨视图在渲染密度、编辑方式和拖拽模型上的真实差异。
- 明确 `chrome mcp` / 自动化测试对交互层的可访问性与可定位要求。
- 明确关系图节点需要满足的 Unreal 蓝图式视觉与端口约束。
- 给出推荐技术路线和不推荐路线，并写清理由。
- 交付一个位于 `BaseLab` 独立应用中的验证实验入口，使用同一批节点、连线和视口指标，同时对比 `DOM + CSS`、`DOM + SVG`、`Canvas` 三种典型方案。
- 验证三种方案在“输出端口拖线预览、输入端口落点连接、视口平移稳定性”上的真实表现。
- 验证三种方案在“端口悬停提示、源端高亮、目标端命中提示”上的可行性、开销和稳定性差异。

## 实施方式

- 先按三视图的核心能力列需求清单，再对候选技术逐项打分。
- 需要把“主交互层是否可以用纯 Canvas”单独作为结论项回答。
- 结论必须能支撑“节点创建、连线、坐标、平移缩放、属性编辑、画布尺寸配置”等能力。
- 验证页作为 `BaseLab` 应用中的 `WBS 3.1 基座对比验证` 实验入口存在，不并入 `Knowledge / Graph / Tracks` 三个正式产品 tab。
- 三种方案必须使用同一组验证节点、同一套拖拽、动态连线与缩放能力，避免因为样例不同导致主观比较失真。
- 端口提示必须统一落到共享状态语义，再由 DOM 和 Canvas 各自映射到不同视觉效果。

## 验收方法

- 人工核验上述设计文档已存在，并至少包含“候选方案”“对比维度”“推荐路线”“例外边界”“放弃原因”五节。
- 人工核验文档已明确回答多视图分支是否共用一种画布技术，以及拖拽、缩放、平移和属性编辑分别依托什么能力实现。
- 人工核验文档已明确回答纯 `Canvas` 对 `chrome mcp` / 自动化测试的影响。
- 人工核验文档已明确写出关系图节点需要具备的蓝图式端口、锚点与连线约束。
- 人工启动 `npm run dev:base-lab` 后，首页应出现 `BaseLab` 标题与 `WBS 3.1 基座对比验证` 实验卡片。
- 点击 `打开 WBS 3.1 基座对比验证` 后，页面中能同时看到 `DOM + CSS`、`DOM + SVG`、`Canvas` 三块验证画布。
- 在三块验证画布中拖动画布时，视口位移应稳定跟随指针，不出现明显漂移。
- 在三块验证画布中从输出端口拖向输入端口时，应能看到动态连线预览，并在落到目标输入端口后形成新连线。
- 在三块验证画布中把鼠标移动到端口或从端口起拖时，应能区分 `hover / source / target` 提示，且提示不会随着轻微移动频繁丢失。
- 在验证页点击 `Back to BaseLab` 后，应返回 `BaseLab` 实验首页。
- 人工启动 `npm run dev:web` 后，主工作台顶部不再出现 `WBS 3.1 Base Lab` 按钮。
