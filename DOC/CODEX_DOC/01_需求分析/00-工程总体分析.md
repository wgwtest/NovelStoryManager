# 工程总体分析

## 当前仓库现状

1. 仓库已具备可运行的前后端与示例数据。
2. 项目级文档根、WBS 主计划、执行契约和验收骨架已经建立。
3. 当前已有产物包括对象库、知识编辑页、关系图、多轨和观察输出的基础实现，以及对应计划文档。
4. 当前新增缺口集中在中高保真原型基线、视图树重排和后续多视图设计分支收束。

## 当前产品边界

根据已确认设计，产品边界如下：

1. 单本小说单项目。
2. Web 形态。
3. 对象库为真源。
4. 左侧 `对象库` 常驻。
5. 当前核心工作面收束为：`卷宗`、`蓝图推演`、`剪辑编排`、`场景调度`。
6. 文件化项目存储。
7. AI 理解与正文抽取完全外置。
8. 章节属于观察维度，不属于底层建模依赖。

## 建议模块分解

### WBS 1. 需求与方案

负责需求边界确认、信息架构、低保真原型、高风险交互验证、技术方案选型、初始任务拆解，以及新增的中高保真原型设计补充节点。

### WBS 2. 核心对象模型与文件化

负责 schema、项目目录、对象读写、校验和导入导出约束。

### WBS 3. 多视图共性机制

负责多种建模工作面共用的视口、渲染基座、元素表示、保存协议与验证基线。

### WBS 4. 视图设计

负责把后续执行分支统一收束到一个视图设计父节点下，再按知识库、关系图和多轨编排三个实现分支继续拆解。

### WBS 7. 观察输出层

负责时间切片、人物切片、地点切片、剧情线切片，以及未来章节维度衔接。

## 主要风险

1. Schema 过早膨胀，导致对象类型和字段失控。
2. 视图状态污染对象事实。
3. 多轨拖拽行为误修改底层事件事实。
4. 还没看清工作台形态就冻结技术栈，后期返工成本高。
5. 关系图与多轨视图都属于强交互页面，如果不先做原型，插件和实现策略容易选错。
6. 如果关系图主交互层退回纯 `Canvas`，`chrome mcp` 与自动化会明显变脆。
7. 如果没有中高保真原型，后续多视图的视觉目标会继续模糊。

## 当前推荐顺序

1. 先固化工程文档根与 WBS。
2. 先完成信息架构与页面分解。
3. 先完成低保真原型设计。
4. 先明确高风险交互验证范围。
5. 再冻结技术方案选型。
6. 再完成初始任务拆解与时间规划。
7. 补入中高保真原型设计与生成方案。
8. 先冻结 `WBS 3` 多视图共性机制。
9. 再进入 `WBS 4` 视图设计下的各执行分支。
