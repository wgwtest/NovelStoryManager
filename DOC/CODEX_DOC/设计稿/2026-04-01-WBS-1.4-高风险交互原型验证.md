# WBS 1.4 高风险交互原型验证

## 目标

在冻结技术栈前，先把真正高风险的拖拽和画布交互拆出来验证，避免只凭“支持拖拽”这类模糊描述选库。

## 本节点需要验证的交互

### 1. 对象库到工作区的拖拽

需要验证：

1. 从左侧对象库拖拽对象到 Graph
2. 从左侧对象库拖拽事件或角色到 Tracks
3. 拖入目标区域时的吸附、占位和取消行为

### 2. Graph 视图交互

需要验证：

1. 节点拖拽
2. 节点连线
3. 边编辑
4. 子图聚焦
5. 缩放与平移

### 3. Tracks 视图交互

需要验证：

1. 事件块拖拽入轨
2. 轨道块跨轨移动
3. 轨道块排序
4. 轨道块吸附
5. 轨道维度切换后布局是否仍清晰

### 4. 通用编辑器交互

需要验证：

1. 多选
2. 框选
3. 键盘删除或移动
4. 右侧检查器联动

## 插件候选与建议

### 通用拖拽

建议候选：`dnd-kit`

理由：

1. 更适合可组合的编辑器级拖拽
2. 便于对象库到轨道区、列表重排和自定义吸附逻辑统一处理

### 关系图

建议候选：`React Flow`

理由：

1. 自带节点拖拽、连线、缩放和平移能力
2. 更接近本项目 Graph 视图需求，不必从零搭图编辑器

### 图布局

建议候选：`elkjs`

理由：

1. 可为复杂关系图提供自动布局
2. 适合在手工布局之外提供“重新整理”入口

### 分栏与面板尺寸调整

建议候选：`react-resizable-panels`

理由：

1. 适合左侧对象库、中心工作区、右侧检查器三栏布局
2. 不需要自己从零处理拖动分栏

### 轨道视图

建议策略：

1. 首版不优先选择通用 timeline 插件
2. 先采用 `dnd-kit + 自定义 DOM 轨道编辑层`
3. 当轨道块数量、缩放密度或性能压力明显上升时，再评估 `Konva` 一类 canvas 方案

理由：

1. 小说轨道编排不是标准甘特图，也不是标准时间轴
2. 过早采用通用 timeline 插件容易被现成交互模型反向限制

## 验证输出要求

1. 每一类高风险交互都要写清是否可行
2. 每一类候选库都要写清采用理由和不采用理由
3. 轨道视图必须单独说明为什么不直接押注通用 timeline 插件
