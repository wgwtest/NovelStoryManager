# WBS 1.2 技术方案选型

## 目标

在进入 `WBS 2` 实现前，先冻结当前阶段最关键的技术决定，让对象库、知识库视图、关系图视图、轨道视图都建立在同一套运行边界上。

## 已选方案

### 1. 前端形态

1. 采用 `React + Vite + TypeScript`
2. 理由：
   - 适合构建左侧常驻对象库和多 tab 工作区
   - 便于后续引入关系图和轨道视图这种交互较强的页面
   - 当前仓库从零开始，采用成熟前端工具链更稳妥

### 2. 本地服务形态

1. 采用 `Fastify + Node.js`
2. 理由：
   - 本项目需要真实访问本地文件系统
   - 文件读写、schema 校验和项目打开/保存应放在同一层处理
   - 前端只负责视图与编辑，不直接碰磁盘

### 3. 数据约束方式

1. 采用 `TypeScript + Zod`
2. 理由：
   - 同时满足运行时校验和类型约束
   - 适合文件导入时做结构化验证
   - 后续外部智能体产出的规格化数据也可直接按 schema 校验

### 4. 仓库组织方式

1. 采用单仓库 `npm workspaces`
2. 初始拆分为：
   - `apps/web`
   - `apps/service`
   - `packages/schema`
3. 理由：
   - 共享 schema 最方便
   - 前后端边界清楚
   - 仍然保持单项目开发成本较低

### 5. 文件化项目结构

1. 对象库作为真源，按对象类型分文件存储
2. 视图状态单独存储，不与对象事实混写
3. 初始建议目录：
   - `manifest.json`
   - `schema-version.json`
   - `objects/*.json`
   - `views/*.json`
4. 理由：
   - 便于导入导出
   - 便于后续用任何智能体生成规格化数据后再载入
   - 便于控制“事实数据”和“展示布局”解耦

### 6. 前后端职责划分

1. `service` 负责：
   - 项目创建
   - 项目打开
   - 项目保存
   - schema 校验
   - 文件系统读写
2. `web` 负责：
   - 左侧对象库常驻
   - 知识库视图
   - 关系图视图
   - 轨道视图
   - 对对象模型的编辑交互
3. 约束：
   - `web` 不直接操作本地文件
   - `service` 不承载“理解正文”的 AI 逻辑

### 7. 测试基线

1. `Vitest` 作为基础测试框架
2. 前端组件测试使用 `React Testing Library`
3. 服务层至少覆盖：
   - 项目打开
   - 项目保存
   - schema 校验失败

## 暂不采用的方案

1. 不采用数据库作为首轮真源
   - 当前明确要求文件化项目优先，数据库会提前引入迁移和部署复杂度
2. 不采用前端直写文件系统
   - 容易把视图交互和存储细节耦合在一起
3. 不在工具内部加入正文 AI 理解链路
   - 与已确认的产品边界冲突

## 对 `WBS 2.1` 的直接约束

1. 首轮实现必须先落 `workspace + schema + service + knowledge view`
2. 关系图视图和轨道视图只保留入口或占位，不在本节点进入复杂交互
3. 所有编辑最终都要落回对象库文件，而不是临时内存模型
4. 章节不是底层建模前提，不得反向决定对象模型
