# 工程总体分析

## 当前仓库现状

1. 仓库刚初始化，尚无业务代码
2. 当前已有产物主要是设计稿
3. 尚未建立项目级文档根、WBS 主计划、执行契约和验收骨架
4. 尚未确定前端技术选型、运行时框架和文件读写承载方式

## 当前产品边界

根据已确认设计，产品边界如下：

1. 单本小说单项目
2. Web 形态
3. 对象库为真源
4. 三个主视图：知识库视图、关系图视图、轨道视图
5. 文件化项目存储
6. AI 理解与正文抽取完全外置
7. 章节属于观察维度，不属于底层建模依赖

## 建议模块分解

### WBS 1. 项目启动与工程基线

负责文档根、计划骨架、技术底座、运行入口和首轮工程规范。

### WBS 2. 核心对象模型与文件化项目

负责 schema、项目目录、对象读写、校验和导入导出约束。

### WBS 3. 知识库主视图

负责左侧对象库常驻、对象列表、详情编辑、字段维护和筛选。

### WBS 4. 关系图工作台

负责对象关系可视化、边编辑和布局保存。

### WBS 5. 多轨编排视图

负责按地点、剧情线、角色视角和势力等维度进行派生编排。

### WBS 6. 观察输出层

负责时间切片、人物切片、地点切片、剧情线切片，以及未来章节维度衔接。

## 主要风险

1. Schema 过早膨胀，导致对象类型和字段失控
2. 视图状态污染对象事实
3. 轨道拖拽行为误修改底层事件事实
4. 还没冻结 schema 就启动大量 UI 开发，后期返工成本高

## 当前推荐顺序

1. 先固化工程文档根与 WBS
2. 先写 M1 详细计划
3. 先实现项目文件骨架与 schema，再做 UI 框架
4. 关系图和轨道视图作为后续阶段推进

## 当前非目标

2. 立即做章节正文系统
