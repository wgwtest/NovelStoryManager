# WBS 2.2 项目数据规范与导入导出规范

## 1. 目标

1. 固化单项目文件存储与对外导入导出之间的稳定映射。
2. 让外部智能体只要按规范生成结构化数据，就能被本工具校验和载入。
3. 保持“对象库是真源、视图状态独立、AI 理解外置”这三条前置约束不被破坏。
4. 为卷宗、多视图与观察输出提供同一套可扩展的数据底座。

## 2. 作用边界

1. 本规范只定义结构化数据长什么样、如何校验、如何进出系统。
2. 本规范不负责正文理解、章节自动解析、关系自动推断。
3. 本规范不把卷宗做成单独的大文件，而是规定卷宗如何从对象真源与视图状态中被聚合出来。
4. 本规范不改变内部真源：内部真源仍然是项目目录下的多文件对象库。

## 3. 总体设计原则

### 3.1 真源与投影分离

1. 对象事实只允许写入 `objects/*.json`。
2. 视图状态、布局状态、投影结果只允许写入 `views/*.json`。
3. 完整度、异常检测、统计分析等结果默认视为派生信息，不应直接污染对象真源。

### 3.2 卷宗是观察模型，不是存储文件

卷宗统一模型可以被描述为：

`元信息 + 摘要 + 属性 + 时间线 + 关系 + 事件 + 证据 + 分析 + 备注`

但这不意味着项目内必须存在一个 `dossier.json`。

更合理的口径是：

1. 卷宗是对象真源、视图状态和派生结果的聚合观察面。
2. 同一对象在卷宗、蓝图、轨道、场景、章节输出中可以有不同投影。
3. 各视图共享数据模型，但不共享页面实现。

### 3.3 AI 理解完全外置

1. 本工具本身不负责从正文中理解语义。
2. 外部智能体可以把正文和抽象规范转成结构化数据。
3. 本工具只负责校验、载入、编辑和导出这些结构化结果。

## 4. 内部项目文件规范

### 4.1 目录结构

项目根目录必须包含：

1. `manifest.json`
2. `schema-version.json`
3. `objects/`
4. `views/`

### 4.2 对象真源目录

`objects/` 下按对象类型分文件存储：

1. `characters.json`
2. `factions.json`
3. `locations.json`
4. `items.json`
5. `realm-systems.json`
6. `events.json`
7. `relations.json`
8. `clues.json`
9. `arcs.json`

### 4.3 视图状态目录

`views/` 下按观察或编排状态分文件存储：

1. `graph-layouts.json`
2. `track-presets.json`
3. `chapter-slices.json`
4. `saved-filters.json`

### 4.4 真源约束

1. 人物、宗门、地点、物品、体系、事件、关系、线索、主线等对象事实只能写入 `objects/*.json`。
2. 图布局、轨道排布、章节切片、筛选器等观察和编排状态只能写入 `views/*.json`。
3. 原始正文、AI 推断过程、临时理解链路不属于本规范的真源文件。

## 5. 卷宗统一观察模型与数据映射

参考资料中“卷宗统一结构”的思路可以吸收，但必须转译成当前项目的数据映射口径。

### 5.1 元信息

当前主要来源：

1. 对象 `id`
2. 对象类型
3. `name`
4. `status`
5. `tags`

后续可扩展但当前 `1.0.0` 非强制：

1. `createdAt`
2. `updatedAt`
3. `reviewStatus`
4. `completeness`

### 5.2 摘要

当前主要来源：

1. `summary`
2. `status`
3. `tags`
4. 个别对象的类型专属定位字段

### 5.3 属性

属性层由对象类型专属字段承载。
例如：

1. 人物：`identity / factionRefs / realmState`
2. 势力：`goal / locationRefs`
3. 地点：`controllerRef / traits`
4. 物品：`ownerRef / origin / traits`
5. 体系：`levels / rules`
6. 事件：`participantRefs / locationRefs / arcRefs / clueRefs`
7. 关系：`sourceRef / targetRef / type / strength / startAnchor / endAnchor`

### 5.4 时间线

当前 `1.0.0` 不单独定义 `timeline-entries` 真源文件。
时间维度优先由以下结构聚合：

1. `events[].timeAnchor`
2. `relations[].startAnchor / endAnchor`
3. `views/chapter-slices.json`

这意味着：

1. 时间线是观察模型中的核心模块
2. 但当前项目中，时间线优先通过事件事实和章节投影来生成
3. 不应急着把时间线私有化成卷宗内部的一段不可复用数组

### 5.5 关系

关系必须是独立关系记录，而不是散落在对象字段中的非结构化链接。

当前强制规范见 `relations.json`，至少包含：

1. `id`
2. `type`
3. `sourceRef`
4. `targetRef`
5. `direction`
6. `strength`
7. `startAnchor`
8. `endAnchor`
9. `summary`
10. `tags`

后续可扩展但当前 `1.0.0` 非强制：

1. `subtype`
2. `status`
3. `sourceChapterRefs`
4. `confidence`
5. `changedAt`

### 5.6 事件

事件层当前由 `events.json` 承担，是连接人物、地点、势力、物品、线索、主线的重要桥梁。

### 5.7 证据

证据是卷宗的重要组成，但当前 `1.0.0` 尚未定义独立证据真源文件。

当前可承载的证据近似来源：

1. `views/chapter-slices.json` 中的章节摘要与正文片段
2. 对象中的 `notes`
3. 未来由外部智能体导入的结构化来源记录

当前设计原则：

1. 证据很重要，但不能为了补证据而把原始正文或 AI 推断链塞进对象真源。
2. 若后续证据成为稳定的一等数据，应在未来版本中新增独立来源命名空间，而不是混入 `views/`。

### 5.8 分析

分析层属于派生信息。
例如：

1. 关系密度
2. 出场热度
3. 章节覆盖
4. 对象完整度
5. 异常检测结果

当前规范要求：

1. 分析结果默认不写回对象真源。
2. 如需落盘，应进入后续专门的派生结果或报告命名空间，而不是混写到对象事实中。

### 5.9 备注与待确认

当前主要来源：

1. 对象中的 `notes`
2. 补录提示与审校队列

后续如出现稳定需求，可扩展为独立批注记录，但当前 `1.0.0` 不强制。

## 6. 当前 1.0.0 强制字段与推荐扩展的分层

为避免“参考资料很完整，但现实实现跟不上”的问题，本规范明确分层：

### 6.1 当前 1.0.0 强制

1. 项目根目录结构
2. 九类对象文件
3. 四类视图文件
4. 对象引用完整性校验
5. 项目级 bundle
6. 对象级 bundle
7. 坏引用、重复对象、版本不兼容快速失败

### 6.2 当前 1.0.0 推荐但不强制

1. 对象尽量提供 `summary`
2. 关系尽量提供 `summary / tags / startAnchor`
3. 事件尽量提供 `timeAnchor`
4. 章节切片尽量提供 `eventRefs / focusObjectRefs`

### 6.3 后续版本扩展预留

1. 证据记录
2. 可信度分级
3. 完整度评分
4. 异常检测结果
5. 更细的对象公共元字段

这些内容可以进入未来 schema 版本，但当前不强行写成 `1.0.0` 的硬约束。

## 7. 时间线与章节切片规范

### 7.1 章节不是建模前提

用户已经明确提出：
`一个小说即使没有章节，也不影响建模。`

因此本规范要求：

1. 章节切片是观察和输出维度
2. 章节切片不是对象事实真源
3. 没有章节时，项目仍可只依赖对象和事件完成建模

### 7.2 当前章节切片结构

`views/chapter-slices.json` 当前至少包含：

1. `id`
2. `title`
3. `summary`
4. `text`
5. `sourceMode`
6. `eventRefs`
7. `focusObjectRefs`
8. `order`

### 7.3 时间锚点与章节投影的区别

1. `timeAnchor` 表达的是事实发生锚点，属于事件或关系事实的一部分。
2. `chapter-slices` 表达的是观察输出和章节展示顺序，属于视图投影。
3. 不能把这两者混成同一层数据。

## 8. 完整度与异常检测的存储边界

参考资料强调完整度和异常检测，这一点是合理的。
但在当前项目里，必须先把边界说清：

### 8.1 完整度

完整度是派生判断，不是对象真源。
它的计算依赖于：

1. 摘要是否存在
2. 关系是否完整
3. 事件是否覆盖
4. 是否存在待确认项
5. 是否存在章节或证据投影

### 8.2 异常检测

异常检测结果同样是派生结果。
例如：

1. 年龄冲突
2. 时间线冲突
3. 关系状态矛盾
4. 场景归属矛盾
5. 物品归属冲突

当前规范要求：

1. 异常检测结果默认不写入对象真源。
2. 后续若需要落盘，应进入独立报告或派生结果命名空间。

## 9. 对外导入导出规范

### 9.1 项目级 bundle

项目级 bundle 用于完整导入或完整导出一个小说项目，建议文件名为 `project-bundle.json`。

建议结构：

```json
{
  "bundleType": "project-bundle",
  "bundleVersion": "1.0.0",
  "exportMeta": {
    "sourceProjectId": "sample-novel",
    "exportedAt": "2026-04-02T00:00:00Z"
  },
  "project": {
    "manifest": {},
    "schemaVersion": {},
    "objects": {},
    "views": {}
  }
}
```

约束：

1. `bundleType` 必须为 `project-bundle`。
2. `bundleVersion` 必须命中当前受支持版本。
3. `project.manifest`、`project.schemaVersion`、`project.objects`、`project.views` 均为必填。
4. `project.objects` 必须满足内部对象集合 schema。
5. `project.views` 必须满足内部视图集合 schema。

### 9.2 对象级 bundle

对象级 bundle 用于结构化增量导入或按对象范围导出，建议文件名为 `object-batch.json`。

建议结构：

```json
{
  "bundleType": "object-batch",
  "bundleVersion": "1.0.0",
  "exportMeta": {
    "sourceProjectId": "sample-novel",
    "exportedAt": "2026-04-02T00:00:00Z"
  },
  "scope": {
    "targetProjectId": "sample-novel",
    "objectTypes": ["characters", "relations"]
  },
  "objects": {
    "characters": [],
    "relations": []
  }
}
```

约束：

1. `bundleType` 必须为 `object-batch`。
2. `scope.targetProjectId` 必须存在。
3. `scope.objectTypes` 必须与 `objects` 内的集合类型一致。
4. 对象级 bundle 默认不携带 `views`，避免把布局状态混入对象事实。
5. 导入时若发现引用对象既不在当前 batch 中，也不在目标项目中，必须失败，不允许静默跳过。

## 10. 校验规则

1. 根字段缺失必须失败。
2. `bundleType` 非法必须失败。
3. `bundleVersion` 不受支持必须失败。
4. 同一对象集合中 `id` 重复必须失败。
5. 关系、事件、线索、剧情弧、章节切片中的引用无法解析时必须失败。
6. 项目级 bundle 导入后导出的结构应与导入前保持语义一致。
7. 对象级 bundle 导入时只允许改动目标对象集合，不得污染未声明对象类型。
8. 对象级 bundle 不得偷偷携带布局、筛选器、轨道等视图状态。

## 11. 版本规则

1. `schema-version.json` 表达项目内部 schema 版本。
2. `bundleVersion` 单独表达导入导出包规范版本。
3. `schemaVersion` 与 `bundleVersion` 任一方不受支持时，都必须快速失败。
4. 当前规范仍以 `1.0.0` 为基础，不在本轮实现自动迁移。
5. 参考资料中提到的证据、可信度、完整度、异常结果等增强字段，若未来成为正式规范，必须通过新的 schema 版本引入，不能偷偷并入现有版本。

## 12. 失败处理规则

1. 校验失败时必须返回明确错误，不允许自动修补数据。
2. 对象级导入遇到坏引用时必须整体失败，不允许部分成功。
3. 版本不兼容时必须返回版本错误，而不是落入通用解析错误。
4. 对象事实与视图状态混写时必须明确报错。

## 13. 非目标

1. 不在本节点内置任何 AI 文本抽取或语义理解能力。
2. 不在本节点把卷宗做成单文件大对象。
3. 不在本节点实现完整度算法、异常检测算法或证据系统的正式落盘。
4. 不在本节点实现轨道、关系图或场景视图的真实交互。
5. 不在本节点把章节视为建模前提。
