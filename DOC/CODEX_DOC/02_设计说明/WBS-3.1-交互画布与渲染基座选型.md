# WBS 3.1 交互画布与渲染基座选型

## 结论

1. `WBS 3.1` 已完成用户人工确认，当前冻结为已验收结论。
2. 后续多视图实现基线采用“`DOM` 优先、`SVG` 辅助”的路线，不采用纯 `Canvas` 作为主交互层。
3. `卷宗` 继续使用 `DOM` 表单与信息分区，不引入单独画布层。
4. `蓝图推演` 使用 `DOM 节点 + SVG 边层 + 共享视口状态` 的混合方案。
5. `剪辑编排` 与 `场景调度` 使用 `DOM` 主交互层，并在需要表达连线、转场或覆盖提示时允许引入 `SVG` 辅助层。
6. 纯 `Canvas` 只允许承担网格、背景或预览等非关键装饰层。
7. `WBS 3.1` 不再只输出选型文字，而是补交一个位于独立 `BaseLab` 应用中的验证实验入口，供人工比较三种典型基座效果。

## 选择理由

1. 用户已经明确选择 `DOM + SVG` 作为后续实现依据，因此本节点的职责从“比较谁更好”切换为“冻结边界与例外”。
2. 关系图需要节点按钮、属性编辑入口、边标签和可选中状态，`SVG + DOM` 更容易落地。
3. 多轨与空间工作面需要高密度拖拽和可观察热点，但主交互仍应保留 `DOM` 语义，避免退化为纯坐标点击。
4. 多视图共享的核心不是同一渲染标签，而是同一套视口、坐标和保存协议。

## 对比验证入口定义

1. 本节点额外交付一个独立入口：`BaseLab` 应用中的 `WBS 3.1 基座对比验证` 实验卡片。
2. 该入口是验证实验，不是产品正式工作面，因此不并入 `卷宗 / 蓝图推演 / 剪辑编排 / 场景调度` 主工作台。
3. 验证页必须同时展示三种典型方案：
   - `DOM + CSS`
   - `DOM + SVG`
   - `Canvas`
4. 三种方案必须使用同一批验证节点、同一组边、同一套视口动作，至少覆盖：
   - 节点拖拽
   - 画布平移
   - 缩放
   - 画布尺寸配置
   - 坐标观察
   - 输出端口动态拖线预览
   - 输入端口落点连接
5. 用户的决策依据应来自“同页对比后的视觉与交互观感”，而不是只读分析文档。

## 端口提示与命中反馈补充结论

1. 本轮补充后，三种验证基座统一使用同一套端口状态语义：
   - `idle`
   - `hover`
   - `source`
   - `target`
2. `DOM + CSS`
   - 可行性：高
   - 实现开销：低
   - 稳定性：高
   - 说明：端口本身就是 DOM 按钮，只需补 `data-port-tone` 与 CSS 高亮即可。
3. `DOM + SVG`
   - 可行性：高
   - 实现开销：低到中
   - 稳定性：高
   - 说明：节点端口仍保留 DOM 语义，端口提示实现与 `DOM + CSS` 基本一致，只需同步 SVG 预览线与目标端高亮。
4. `Canvas`
   - 可行性：中到高
   - 实现开销：中
   - 稳定性：中
   - 说明：需要在 `pointermove` 中做端口命中检测，并通过重绘光环和端口描边表达提示；缩放、平移或命中半径处理不当时更容易抖动。
5. 本轮只补“端口可发现性”和“命中可见性”，不在当前节点扩展无效目标兼容矩阵，避免把验证页提前做成完整蓝图编辑器。

## 推荐路线

1. 抽出共享 `viewport` 概念：
   - `zoom`
   - `offsetX`
   - `offsetY`
   - `canvasWidth`
   - `canvasHeight`
2. `蓝图推演`、`剪辑编排` 与 `场景调度` 都保存该视口状态。
3. `卷宗` 不保存画布状态，但共享对象选择与对象编辑链路。
4. `蓝图推演` 继续沿用当前已落地的 `DOM 节点 + SVG 边层` 路线，后续只增强节点与端口表达，不切回纯 `Canvas`。
5. 当前推荐不再是“暂定”，而是已冻结为后续实现基线：`DOM` 优先，必要处辅以 `SVG`。
6. 当前这套 `DOM 节点 + SVG 边层` 的组合可以实现接近 Unreal 蓝图节点的视觉与交互效果：
   - 节点主体、标题条、参数区和端口热点保留为 DOM
   - 连线、贝塞尔曲线和吸附反馈由 SVG 负责
   - 因此既能做出蓝图式节点头、输入输出端口与连线层次，也不会丢掉自动化可定位性
7. 共享视口状态的平移计算必须按“初始 offset + 当前指针位移”求值，不能在连续 move 中累计漂移。

## Unreal 蓝图节点约束

1. 关系图节点不是普通矩形卡片，而是“可编辑节点”。
2. 节点至少包含：
   - 标题条
   - 节点类型或分类色带
   - 左侧输入端口列
   - 右侧输出端口列
   - 参数区 / 字段区
   - 选中、悬停、脏状态提示
3. 端口必须有可见锚点，连线时有吸附反馈。
4. 端口在鼠标移动、聚焦或连线过程中，必须能区分 `hover / source / target` 三种可见提示。
5. 连线必须区分有效连接、无效连接和当前悬停目标。
6. 这些约束会进一步落入 `WBS 4.2.1` 的蓝图推演要素定义，不只停留在渲染层说明。
7. 结论：这套模式可以达到“模仿 Unreal 引擎蓝图节点效果”的目标，且比纯 Canvas 更适合当前阶段的可测性要求。

## 放弃路线

1. 不使用纯 `Canvas`：
   - 当前阶段不利于表单交互与可测性。
2. 不使用重型图编辑器/时间轴插件：
   - 容易被插件交互模型绑死，不利于“对象真源 / 视图状态分离”。

## 直接约束

1. 图和轨道必须支持平移、缩放和尺寸配置。
2. 节点与事件块必须能显示坐标或锚点信息。
3. 交互编辑改模型对象时走对象接口，改布局时走视图状态接口。
4. 关系图节点与轨道块的关键交互热点必须保持为 DOM 元素。
5. 基座对比验证入口必须保持为独立观察面，现阶段通过 `BaseLab` 独立应用承载，不占用正式产品主工作面的长期位置。
6. `BaseLab` 的三种验证基座都必须能实际演示“端口起拖 -> 连线预览 -> 目标端口落点连接”的流程，不能只保留静态示意线。
7. `BaseLab` 的三种验证基座都必须在端口悬停、点击或连线经过时，给出明显的命中提示和源/目标高亮。
