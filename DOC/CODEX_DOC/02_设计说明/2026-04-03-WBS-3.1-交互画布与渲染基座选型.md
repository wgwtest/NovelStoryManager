# WBS 3.1 交互画布与渲染基座选型

## 结论

1. `Knowledge` 继续使用 DOM 表格和表单布局，不引入单独画布技术。
2. `Graph` 使用 `DOM 节点 + SVG 边层 + 共享视口状态` 的混合方案。
3. `Tracks` 使用 `DOM 轨道层 + 共享视口状态` 的方案，不采用现成 timeline 插件。
4. 纯 `Canvas` 不作为主交互层，只允许承担网格、背景或预览等非关键装饰层。
5. `WBS 3.1` 不再只输出选型文字，而是补交一个位于独立 `BaseLab` 应用中的验证实验入口，供人工比较三种典型基座效果。

## 选择理由

1. 本项目对象数量在当前阶段不大，DOM 可维护性优先于纯 `Canvas`。【我认为后续对象数量会非常大】
2. 关系图需要节点按钮、属性编辑入口、边标签和可选中状态，`SVG + DOM` 更容易落地。
3. 多轨视图需要轨道头、块、章节观察区和表单交互，DOM 更适合。
4. 三视图共享的核心不是同一渲染标签，而是同一套视口、坐标和保存协议。

## 对比验证入口定义

1. 本节点额外交付一个独立入口：`BaseLab` 应用中的 `WBS 3.1 基座对比验证` 实验卡片。
2. 该入口是验证实验，不是产品正式 tab，因此不并入 `Knowledge / Graph / Tracks` 主工作台。
3. 验证页必须同时展示三种典型方案：
   - `DOM + CSS`
   - `DOM + SVG`
   - `Canvas`
4. 三种方案必须使用同一批验证节点、同一组边、同一套视口动作，至少覆盖：
   - 节点拖拽
   - 画布平移
   - 缩放
   - 画布尺寸配置
   - 坐标观察
5. 用户的决策依据应来自“同页对比后的视觉与交互观感”，而不是只读分析文档。

## 推荐路线

1. 抽出共享 `viewport` 概念：
   - `zoom`
   - `offsetX`
   - `offsetY`
   - `canvasWidth`
   - `canvasHeight`
2. `Graph` 与 `Tracks` 都保存该视口状态。
3. `Knowledge` 不保存画布状态，但共享对象选择与对象编辑链路。
4. `Graph` 继续沿用当前已落地的 `DOM 节点 + SVG 边层` 路线，后续只增强节点与端口表达，不切回纯 `Canvas`。
5. 当前推荐仍是 `DOM + SVG`，但在用户完成验证页人工确认前，只视为“暂定推荐”，不把它写死为不可变结论。
6. 当前这套 `DOM 节点 + SVG 边层` 的组合可以实现接近 Unreal 蓝图节点的视觉与交互效果：
   - 节点主体、标题条、参数区和端口热点保留为 DOM
   - 连线、贝塞尔曲线和吸附反馈由 SVG 负责
   - 因此既能做出蓝图式节点头、输入输出端口与连线层次，也不会丢掉自动化可定位性

## Unreal 蓝图节点约束

1. 关系图节点不是普通矩形卡片，而是“可编辑节点”。
2. 节点至少包含：
   - 标题条
   - 节点类型或分类色带
   - 左侧输入端口列
   - 右侧输出端口列
   - 参数区 / 字段区
   - 选中、悬停、脏状态提示
3. 端口必须有可见锚点，连线时有吸附反馈。
4. 连线必须区分有效连接、无效连接和当前悬停目标。
5. 这些约束会进一步落入 `WBS 5.1` 的关系图要素定义，不只停留在渲染层说明。
6. 结论：这套模式可以达到“模仿 Unreal 引擎蓝图节点效果”的目标，且比纯 Canvas 更适合当前阶段的可测性要求。

## 放弃路线

1. 不使用纯 `Canvas`：
   - 当前阶段不利于表单交互与可测性。
2. 不使用重型图编辑器/时间轴插件：
   - 容易被插件交互模型绑死，不利于“对象真源 / 视图状态分离”。

## 直接约束

1. 图和轨道必须支持平移、缩放和尺寸配置。
2. 节点与事件块必须能显示坐标或锚点信息。
3. 交互编辑改模型对象时走对象接口，改布局时走视图状态接口。
4. 关系图节点与轨道块的关键交互热点必须保持为 DOM 元素。
5. 基座对比验证入口必须保持为独立观察面，现阶段通过 `BaseLab` 独立应用承载，不占用正式产品 tab 的长期位置。
