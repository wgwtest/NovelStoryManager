# WBS 3.1 交互画布与渲染基座选型

## 决策摘要

1. `Knowledge` 继续使用 DOM 列表与表单布局，不引入单独画布技术。
2. `Graph` 使用 `DOM 节点 + SVG 连线层 + 共享视口状态` 的混合方案。
3. `Tracks` 使用 `DOM 轨道层 + 共享视口状态` 的方案，不采用现成 timeline 插件。
4. 纯 `Canvas` 不作为主交互层，只允许承担网格、背景或预览等非关键装饰层。

## 候选方案对比

### DOM

优点：

1. 按钮、输入框、节点端口、属性入口天然可交互。
2. 更利于可访问性、浏览器自动化和 `chrome mcp` 定位。
3. 当前仓库已经在 `GraphView` 和 `TracksView` 中落了 DOM 视口与尺寸配置。

不足：

1. 复杂图形需要额外组合 `SVG` 或绝对定位图层。

### SVG

优点：

1. 非常适合关系边、锚点、高亮路径与可视化连线。
2. 与 DOM 节点混合时，调试与自动化依然可控。

不足：

1. 不适合独立承担复杂表单节点内部的全部交互。

### Canvas

优点：

1. 适合高密度背景绘制、网格、缩略图与大面积装饰渲染。
2. 在超大规模元素渲染时有性能潜力。

不足：

1. 节点、端口、工具栏、尺寸输入和属性入口如果都压到 `Canvas`，语义会明显变差。
2. 自动化会退化成坐标点击，定位与回归成本上升。
3. 当前阶段真实瓶颈并不是绘制性能，而是交互清晰度与可维护性。

## `chrome mcp` 与自动化约束

1. 本项目当前的浏览器自动化与 `chrome mcp` 工作流，本质上依赖 DOM 与可访问性树快照。
2. 如果主交互层改成纯 `Canvas`，节点、端口、连线锚点和工具按钮会丢失元素级语义，自动化只能依赖屏幕坐标，稳定性明显下降。
3. 因此以下要素必须保持为真实 DOM 元素：
   - 节点卡片
   - 端口 / 锚点
   - 节点工具按钮
   - 轨道块
   - 缩放与尺寸输入
   - 右侧属性入口
4. `Canvas` 如需使用，只能承担背景网格、热区预览、缩略图或不影响关键操作的视觉层。

## 推荐路线

1. 抽出共享 `viewport` 概念：
   - `zoom`
   - `offsetX`
   - `offsetY`
   - `canvasWidth`
   - `canvasHeight`
2. `Graph` 与 `Tracks` 都保存该视口状态。
3. `Knowledge` 不保存画布状态，但共享对象选择与对象编辑链路。
4. `Graph` 继续沿用当前已落地的 `DOM 节点 + SVG 边层` 路线，后续只增强节点与端口表达，不切回纯 `Canvas`。

## Unreal 蓝图节点约束

1. 关系图节点不是普通矩形卡片，而是“可编辑节点”。
2. 节点至少包含：
   - 标题条
   - 节点类型或分类色带
   - 左侧输入端口列
   - 右侧输出端口列
   - 参数区 / 字段区
   - 选中、悬停、脏状态提示
3. 端口必须有可见锚点，连线时有吸附反馈。
4. 连线必须区分有效连接、无效连接和当前悬停目标。
5. 这些约束会进一步落入 `WBS 5.1` 的关系图要素定义，不只停留在渲染层说明。

## 放弃路线

1. 不使用纯 `Canvas` 作为主交互层。
2. 不使用重型图编辑器/时间轴插件作为唯一心智模型。

原因：

1. 它们会削弱可测性与元素级可维护性。
2. 它们容易把本项目反向绑到插件自己的数据模型上。

## 直接约束

1. 图和轨道必须支持平移、缩放和尺寸配置。
2. 节点与事件块必须能显示坐标或锚点信息。
3. 交互编辑改模型对象时走对象接口，改布局时走视图状态接口。
4. 关系图节点与轨道块的关键交互热点必须保持为 DOM 元素。
