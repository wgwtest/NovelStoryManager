# WBS 1.8 BaseLab验证工具构建方案

## 目标

在现有 `WBS 3.1 Base Lab` 验证页基础上，定义一套可跨节点复用的验证工具方案，使后续技术选型、交互机制、要素表达和样式实验都能在独立入口中进行，而不污染主工程工作台。

## 问题定义

1. 当前验证页已经证明“可视化对比验证”是有价值的。
2. 但它仍嵌在主工作台入口里，后续如果继续扩展，会逐步污染正式产品的顶层结构。
3. 后续可复用的验证场景并不只存在于 `WBS 3.1`：
   - `WBS 5.1` 关系图节点与端口机制验证
   - `WBS 6.1` 多轨块表达与时间尺验证
   - `WBS 7.x` 观察切片表达与排版验证
4. 因此需要一个比“单页验证页”更稳定的工具化方案。

## WBS 挂载位置判断

### 为什么不是继续挂在 `WBS 3.1`

1. `WBS 3.1` 只负责“交互画布与渲染基座选型”。
2. BaseLab 工具化后将服务多个节点，不应继续归属于单一选型节点。

### 为什么不是直接挂在 `WBS 2`

1. 用户本轮明确要求“先不要立刻编码”。
2. 当前轮次产物是工具构建方案，不是工具底座实现。
3. 因此本轮仍属于“需求与方案”的补充设计阶段。

### 结论

1. 当前新增节点应挂在 `WBS 1`。
2. 节点名称使用直白表达：`WBS 1.8 BaseLab验证工具构建方案`。

## 候选方案对比

### 方案 A：继续放在 `apps/web`，但改为独立路由

优点：

1. 改造成本最低。
2. 可直接复用现有主应用样式与构建配置。

不足：

1. 仍与主产品入口强耦合。
2. 实验页面和产品页面会持续共享依赖、导航与分析上下文。
3. 后续实验节点越多，`apps/web` 越容易变成“产品 + 实验混合体”。

### 方案 B：在同一 monorepo 中新增独立应用 `BaseLab`

优点：

1. 入口、路由、页面代码与主产品明确分离。
2. 仍可复用 monorepo 的 `schema`、样例数据和基础工具函数。
3. 便于后续按实验主题持续增加入口，而不污染主产品信息架构。
4. 与用户强调的“独立、高效、不污染主工程”目标最一致。

不足：

1. 初始会增加一个独立应用壳与少量工程配置。
2. 需要明确哪些共享能力允许复用，哪些产品组件禁止直接引用。

### 方案 C：在仓库外独立建立实验工程

优点：

1. 隔离最彻底。

不足：

1. 会割裂当前 WBS 文档、样例数据、共享代码和验证证据。
2. 会显著增加维护成本，不利于当前迭代速度。

## 推荐路线

### 主推荐

采用方案 B：在当前 monorepo 中新增独立应用 `BaseLab`。

### 推荐原因

1. 既能保持工程隔离，又不会丢失 monorepo 的复用效率。
2. 能把“验证工具”明确建模成独立工作对象，而不是产品页面的附属物。
3. 后续每个验证实验都可以作为 `BaseLab` 下的一个入口，符合用户希望的“统一入口”诉求。

## BaseLab 结构方案

### 入口结构

1. 独立应用目录：
   - `apps/base-lab/`
2. 根入口职责：
   - 展示实验入口列表
   - 按 `WBS` 节点或验证主题分组
   - 标明实验目标、状态和关联文档
3. 实验页路由建议：
   - `/labs/base-selection`
   - `/labs/graph-node-blueprint`
   - `/labs/tracks-density`
   - `/labs/observation-layout`

### 实验注册机制

1. 每个实验使用统一注册对象描述：
   - `id`
   - `title`
   - `targetWbs`
   - `purpose`
   - `entryPath`
   - `sampleDataRefs`
   - `status`
2. 根入口只读取注册表，不直接耦合每个实验实现细节。

### 与主工程的边界规则

1. `BaseLab` 可以复用：
   - `packages/schema`
   - 样例数据
   - 与视口、坐标、基础绘制相关的低层公共工具
2. `BaseLab` 不应直接复用：
   - `apps/web/src/App.tsx`
   - 主工作台的页面级组件
   - 为产品页面定制的复杂状态编排
3. 如果确有共用价值，应先抽到共享包，再同时供主工程与 `BaseLab` 使用。

## 第一批实验范围建议

1. `WBS 3.1` 当前已有的基座对比实验，作为第一批迁移对象。
2. `WBS 5.1` 关系图蓝图节点实验：
   - 节点头部
   - 输入输出端口
   - 吸附点
   - 连线反馈
3. `WBS 6.1` 多轨块表达实验：
   - 轨道头
   - 时间尺
   - 块密度
   - 缩放行为

## 迁移原则

1. 现有 `WBS 3.1 Base Lab` 不立即删除。
2. 在正式批准方案并进入编码前，它仍作为“已形成证据”的临时验证实现存在。
3. 后续若实施 `BaseLab`，应先迁移 `WBS 3.1` 实验，再决定是否从主工作台入口移除旧入口。

## 非目标

1. `BaseLab` 不是第二套正式产品界面。
2. `BaseLab` 不承担正式对象编辑、正式持久化和正式发布职责。
3. `BaseLab` 不应变成任意临时 demo 的堆放区，所有实验入口都必须挂靠明确的 `WBS` 节点。

## 结论

1. 将当前验证页工具化、独立化是可行的。
2. 最合适的当前节点是 `WBS 1.8 BaseLab验证工具构建方案`。
3. 推荐后续进入编码时，在 monorepo 中新增独立应用 `apps/base-lab/`，而不是继续扩展主工作台根入口。
