# WBS 4 多视图设计

## 设计目标

`WBS 4` 的目标不是再定义一套新的对象模型，而是在既有项目数据模型之上，建立多个彼此独立、但由同一真源驱动的工作面。

当前首批确定的工作面为：

1. `卷宗`
2. `蓝图推演`
3. `剪辑编排`
4. `场景调度`

这里必须明确：

1. `WBS 4` 是“多视图设计”父节点，不是“系统最终只剩四个视图”
2. 当前四个工作面只是首批实现对象
3. 后续如有新的观察维度或编辑工作面，可以继续挂到同一父节点之下

用户关注的核心不是“首批四个页面长得像不像一家人”，而是：

1. 它们是否由同一套项目对象驱动
2. 它们是否能独立设计、独立替换
3. 它们是否只通过模型层协作，而不是在视图层互相穿透

本节点的设计结论是：多个工作面都必须严格遵循“同一模型，多视图投影”的结构，当前首批四个工作面先按这一规则落地。

## 真源与分层

### 1. 模型真源

当前项目真源固定为文件化项目目录中的两大层：

1. `objects/*.json`
   - 存放人物、势力、地点、物品、境界体系、事件、关系、线索、剧情弧等对象事实
2. `views/*.json`
   - 存放布局、过滤、轨道预设、章节切片等观察状态与视图状态

其中：

1. 对象事实不允许写回 `views/*.json`
2. 布局、筛选、轨道编排等观察结果不允许反写为对象事实

这意味着多个工作面共享同一套对象库真源，但每个工作面对真源的“观察方式”和“局部布局状态”可以完全不同。

### 2. 视图状态不是对象事实

多个工作面都允许产生各自的视图状态，例如：

1. 当前过滤器
2. 视口位置
3. 节点布局
4. 轨道预设
5. 章节切片
6. 场景舞台布局

这些状态属于“怎么看”和“怎么摆”，不属于“世界本身是什么”。

因此，用户在任何工作面中拖拽、排序、切换布局，本质上都是在修改视图状态对象，而不是直接篡改对象真源。

## MVC 约束

### 1. Model

`Model` 层负责承载项目真源与视图状态真源，至少包括：

1. `objects/*.json` 中的对象数据
2. `views/*.json` 中的视图状态
3. 对象读写、引用校验、导入导出、版本校验等模型规则

模型层不关心“卷宗长什么样”“轨道是什么颜色”，只负责：

1. 数据是否存在
2. 关系是否成立
3. 写回是否合法
4. 一个对象是否可以被不同工作面读取

### 1.1 模型增量补齐策略

本项目不假设在初期一次性设计出“完美且最终定稿”的 `Model`。

随着 `WBS 4` 各工作面推进，若出现新的稳定数据需求，可以增量补齐模型层结构，但必须同时满足以下约束：

1. 新增对象事实仍写入 `objects/*.json`
2. 新增观察状态仍写入 `views/*.json`
3. 不能把某个工作面的局部临时状态伪装成全局对象事实
4. 补齐后的结构必须同步更新数据规范、schema、样例数据和测试
5. 任何新增字段都必须能被解释为显式建模结果，而不是工具内置的隐式 AI 推断

换句话说，本项目允许“增量补模”，但不允许“边做边破坏真源边界”。

### 2. Controller

`Controller` 层负责把用户动作翻译成对模型和视图状态的修改，例如：

1. 新建对象
2. 编辑字段
3. 创建引用
4. 更新节点位置
5. 变更轨道顺序
6. 保存过滤器
7. 生成章节切片

控制器负责“动作语义”，不负责“渲染细节”。

这保证：

1. 同一种对象编辑动作可以被不同工作面复用
2. 工作面之间共享的是动作契约，而不是界面组件

### 3. View

`View` 层负责把同一份数据投影为不同工作面：

1. `卷宗` 把对象投影为档案与事实
2. `蓝图推演` 把对象投影为节点与连接
3. `剪辑编排` 把对象投影为轨道与片段
4. `场景调度` 把对象投影为空间节点与转场

每个工作面可以拥有完全不同的视觉语言、交互模式和布局策略，只要它仍然通过控制器读写同一套模型。

### 4. 严格解耦要求

本项目对四个工作面的解耦要求如下：

1. 不允许 `卷宗` 直接调用 `蓝图推演` 的内部组件或私有状态
2. 不允许 `剪辑编排` 直接依赖 `场景调度` 的布局实现
3. 不允许把某个工作面的局部布局硬编码为全局数据事实
4. 允许共享对象库、检查器、选择状态语义、拖拽契约和视口工具函数
5. 允许共享公共外壳，但不允许共享导致视图互锁的内部实现

换句话说，四个工作面可以共享“语言”和“协议”，但不能共享“彼此的身体”。

## 扩展性约束

`WBS 4` 的扩展性要求如下：

1. 新工作面可以继续加入 `WBS 4` 子树，但必须复用同一对象真源和控制契约
2. 新工作面可以拥有全新布局、全新视觉语言和全新局部工具，不需要模仿当前首批四个页面
3. 新工作面如果需要新增模型字段，必须按“模型增量补齐策略”同步更新规范与测试
4. `WBS 4` 的设计文档和计划文档都应使用“多视图、当前首批四个工作面”的口径，不再使用封闭的“四视图系统”表述

## 共享部分与独立部分

### 1. 允许共享的部分

1. 左侧常驻对象库
2. 右侧检查器外壳
3. 选中对象、当前焦点对象、当前编辑对象的语义
4. 拖拽和落点的基础事件契约
5. 视口与坐标工具
6. 读写项目文件的模型服务

### 2. 必须独立的部分

1. 中央主工作面的页面结构
2. 每个工作面的布局状态
3. 每个工作面的视觉表达
4. 每个工作面的核心交互流程
5. 每个工作面的专有验证标准

因此，“共享 Shell，独立 Workface”是 `WBS 4` 的基本边界。

## 当前首批四个工作面的职责边界

### 1. 卷宗

作用是管理对象事实、字段完整度、引用回查和对象审校。

它回答的是：

1. 这个对象是什么
2. 这个对象有哪些事实
3. 这个对象与谁有关
4. 这份对象资料是否足够完整

它不负责推演关系图，也不负责时间编排和空间调度。

### 2. 蓝图推演

作用是把对象与规则投影为可推演的节点网络。

它回答的是：

1. 这个事件如何触发下一个事件
2. 这个关系如何传播
3. 这条链路能不能连通
4. 当前局部子图在整体推演里位于哪里

它不负责替代卷宗做字段总览，也不负责承载多轨编排。

### 3. 剪辑编排

作用是把对象、事件和剧情块投影为多轨剪辑式编排结果。

它回答的是：

1. 哪些片段被放进哪条线
2. 时间、空间和剧情线如何并列推进
3. 当前章节或切片覆盖了哪些内容
4. 调整顺序后会生成什么观察结果

它不负责空间拓扑，也不负责对象事实核对。

### 4. 场景调度

作用是把地点、驻留对象和转场关系投影为空间组织图。

它回答的是：

1. 谁在什么地点
2. 场景如何在空间上切换
3. 哪些地点发生冲突
4. 某段叙事的空间路径是什么

它不负责时间切片编排，也不负责节点逻辑推演。

## 子视图关系

每个工作面内部都可以再拆为多个子区域，例如：

1. 左侧对象库
2. 中央主区
3. 右侧检查器
4. 中央主区内部的辅助面板、工具栏、状态条

这些子区域之间的关系也必须遵循同一原则：

1. 通过控制器共享选中状态、焦点状态和视图状态
2. 不通过直接互调来传递业务含义
3. 某个子面板被替换后，不应导致其他工作面失效

例如：

1. 卷宗中的“引用回查条”可以被替换为抽屉或底栏
2. 蓝图推演中的“工具栏”可以被换成悬浮条
3. 剪辑编排中的“片段篮”可以被换成侧栏
4. 场景调度中的“调度表”可以被换成列表栏

只要控制器契约不变，视图替换就是允许的。

## 数据驱动矩阵

### 1. 卷宗

1. 主要读：`objects/*.json`
2. 辅助读：`views/saved-filters.json`
3. 主要写：对象字段、过滤器与浏览状态

### 2. 蓝图推演

1. 主要读：`objects/*.json`
2. 辅助读：`views/graph-layouts.json`
3. 主要写：节点布局、视口状态、局部图过滤与选中状态

### 3. 剪辑编排

1. 主要读：`objects/*.json`
2. 辅助读：`views/track-presets.json`、`views/chapter-slices.json`
3. 主要写：轨道预设、切片结果与编排视图状态

### 4. 场景调度

1. 主要读：`objects/locations.json` 及相关对象文件
2. 辅助读：场景布局与调度状态类 `views/*.json`
3. 主要写：场景布局、空间过滤与调度观察状态

## 对后续实现的直接约束

1. 任何一个工作面都只能通过模型层和控制器层影响其他工作面的可见结果。
2. 任何一个工作面的布局与观察状态都不得写回对象事实文件。
3. 共享组件只能沉淀为外壳、协议和基础交互，不得演化成工作面之间的隐藏耦合。
4. `WBS 4` 当前首批四个工作面不是上限，后续扩展工作面仍必须服从本设计约束。
5. 视图开发过程中允许增量补齐迫切需要的模型结构，但必须同步更新 `WBS 2` 相关规范、schema 与测试。
6. `WBS 4.1` 到 `WBS 4.4` 必须先补足父级设计说明，再进入能力验证和功能验证。
7. 当前阶段优先推进 `WBS 4.1 卷宗` 分支，其中 `4.1.2 / 4.1.3 / 4.1.4` 先在 `BaseLab` 用模拟数据验证，之后再进入主工作台融合。
8. `WBS 7` 观察输出层应消费这些工作面的观察结果，但不应反向成为它们的实现前提。
