# WBS 4.1 卷宗工作面设计

## 1. 工作面定义

`卷宗` 不是普通资料页，也不是单纯的人物卡片集合。

这里的卷宗应被定义为：

`围绕某个对象建立的、可持续补录、可回查来源、可关联分析、可直接修订事实的对象档案工作面`

因此卷宗不是视觉风格，而是信息组织方式。
它强调的是：

1. 以对象为中心
2. 以结构化事实为主
3. 以关系和变化为关键维度
4. 以证据与校验提高可信度
5. 以跨视图复用作为事实入口

当前阶段，卷宗也是 `WBS 4` 的优先执行分支，因为它最直接承接对象事实核对、字段补录、关系维护和引用回查，是后续蓝图、编排、场景、观察输出的共同事实入口。

## 2. 卷宗要解决的问题

卷宗工作面至少要解决以下六类问题：

1. 对象事实分散，作者无法集中核对。
2. 对象有静态设定，但缺少动态变化脉络。
3. 关系存在于脑中或正文里，但缺少可维护的关系记录。
4. 对象参与过哪些事件、在哪些章节或切片中出现过，难以回查。
5. 不同视图都会使用同一对象，但缺少统一的事实入口。
6. 后续异常检测、完整度评估和证据回溯缺少稳定承载面。

因此卷宗不是其他工作面的附属说明页，而是所有对象事实的主入口。

## 3. 与其他工作面的分工

卷宗必须与其他工作面严格解耦。

### 3.1 卷宗负责什么

1. 对象浏览与搜索
2. 对象核心字段编辑
3. 对象引用字段编辑
4. 独立关系对象编辑
5. 对象参与事件与章节投影回查
6. 缺失字段、待确认项、完整度与冲突提示观察

### 3.2 卷宗不负责什么

1. 不负责蓝图推演中的结构试排和因果试排
2. 不负责多轨时间线的编排和拖拽
3. 不负责空间舞台的摆位和调度
4. 不负责章节正文主编辑器
5. 不负责 AI 理解、抽取和自动推断

换句话说：

- 卷宗负责定义和修订 `事实是什么`
- 蓝图、轨道、场景负责表达 `事实如何被投影和组织`

## 4. 卷宗不是只读页，而是事实层读写工作面

本轮评阅反复追问的关键问题是：
`卷宗到底是只读资料查阅页，还是一个真正可工作的编辑界面。`

这里明确口径：

1. 卷宗不是只读资料页。
2. 卷宗是对象事实层的读写混合工作面。
3. 但卷宗也不是吞掉所有视图职责的超级编辑器。

### 4.1 卷宗负责的编辑

1. 名称、摘要、状态、标签、身份等核心字段
2. 所属势力、参与事件、关联对象等引用字段
3. 独立关系对象，例如 `sourceRef / targetRef / type / strength / summary / tags`
4. 待确认项、缺失项和备注类信息

### 4.2 卷宗只负责观察、不直接负责排布的部分

1. 时间线总览
2. 章节切片映射
3. 关系网络图
4. 完整度评分
5. 异常检测结果

这些内容在卷宗中应该可见、可回查、可跳转，但不一定都在卷宗内直接完成重编辑。

## 5. 卷宗的统一抽象模型

参考资料中最有价值的部分，是把卷宗从“页面区块”提升到“统一观察模型”。

本项目吸收后的口径如下：

`卷宗 = 元信息 + 摘要 + 属性 + 时间线 + 关系 + 事件 + 证据 + 分析 + 备注`

但这是一种：

`卷宗观察模型`

不是要求项目目录里存在一个巨大的 `dossier.json`。

### 5.1 元信息

用于快速识别对象：

1. 卷宗 ID
2. 对象类型
3. 标题
4. 状态
5. 标签
6. 最近更新时间
7. 完整度或异常状态摘要

### 5.2 摘要

用于快速说明对象当前在小说中的意义：

1. 一句话概述
2. 当前叙事定位
3. 当前阶段状态
4. 最值得关注的变化点

### 5.3 属性

用于承载该对象类型的结构化事实字段。
这是卷宗的基础层，也是最直接的编辑面。

### 5.4 时间线

用于说明对象经历过什么变化。
在当前项目里，时间线不应被理解成卷宗私有数组，而应优先由以下事实聚合而成：

1. 事件对象中的 `timeAnchor`
2. 关系对象中的 `startAnchor / endAnchor`
3. 章节切片中的时间或观察投影

### 5.5 关系

用于承载对象与其他对象之间的结构化连接。
关系不应只是“跳转链接”，而应是可描述、可编辑、可追踪状态变化的关系记录。

### 5.6 事件

用于承载对象参与过或受影响的事件。
它回答的问题不是“这个对象是什么”，而是“这个对象在故事里干了什么、被什么影响了”。

### 5.7 证据

用于说明卷宗中的信息来自哪里。
证据是卷宗区别于普通设定卡片的重要能力，因为它保证事实可追溯。

### 5.8 分析

用于展示派生结果，而不是存放事实真源。
例如：

1. 关系密度
2. 出场热度
3. 关键事件覆盖
4. 完整度
5. 异常检测摘要

### 5.9 备注

用于承载：

1. 待确认事项
2. 作者批注
3. 未来补录计划
4. 当前不宜写成正式事实的临时判断

## 6. 卷宗统一模型与当前数据层的映射

卷宗模型不能破坏当前项目“对象真源 / 视图状态 / 派生结果”三层分离。

### 6.1 对象真源

对象事实仍然只允许落在 `objects/*.json`。

当前主要包括：

1. `characters.json`
2. `factions.json`
3. `locations.json`
4. `items.json`
5. `realm-systems.json`
6. `events.json`
7. `relations.json`
8. `clues.json`
9. `arcs.json`

### 6.2 视图状态

观察投影和布局状态只允许落在 `views/*.json`。

当前主要包括：

1. `graph-layouts.json`
2. `track-presets.json`
3. `chapter-slices.json`
4. `saved-filters.json`

### 6.3 派生结果

完整度、异常检测、热度统计、关系密度等都属于派生结果。
它们不能粗暴写回对象真源，否则会污染事实层。

### 6.4 外部结构化输入

AI 文本理解仍然完全外置。
本工具只接收已经被外部智能体抽象好的结构化数据，不在卷宗内承担任何语义理解责任。

## 7. 当前九类对象的卷宗关注重点

本项目当前比参考资料更细，已经把卷宗对象拆成九类。

### 7.1 人物

重点关注：

1. 身份与状态
2. 所属势力与修炼体系
3. 关键关系
4. 人生轨迹与参与事件
5. 章节切片中的出场情况

### 7.2 势力 / 宗门

重点关注：

1. 组织定位
2. 目标与状态
3. 核心人物
4. 领地与关联地点
5. 与其他势力或人物的关系

### 7.3 地点 / 场景

重点关注：

1. 空间属性
2. 控制者与归属
3. 关联事件
4. 常驻或高频出现人物
5. 状态变化

### 7.4 物品

重点关注：

1. 核心属性
2. 来源
3. 当前归属
4. 流转轨迹
5. 关联事件与人物

### 7.5 体系 / 规则

重点关注：

1. 层级结构
2. 规则定义
3. 与角色状态的对应关系
4. 与世界设定的兼容性

### 7.6 事件

重点关注：

1. 背景
2. 参与对象
3. 发生地点
4. 结果与影响
5. 与主线、线索、章节切片的映射

### 7.7 关系

重点关注：

1. 起点对象
2. 终点对象
3. 关系类型
4. 强度
5. 建立或变化锚点
6. 说明与状态

### 7.8 线索

重点关注：

1. 线索描述
2. 触发条件
3. 关联对象
4. 关联事件
5. 当前揭示状态

### 7.9 主线

重点关注：

1. 主线定位
2. 涉及对象
3. 关键事件
4. 当前推进状态
5. 与章节和观察切片的映射

## 8. 页面结构设计

卷宗页面不应只有“目录 + 表单”。

### 8.1 左侧对象库

负责：

1. 搜索对象
2. 按类型切换
3. 按标签、状态、完整度、异常状态筛选
4. 使用已保存筛选器
5. 进入具体卷宗

目录条目建议至少可显示：

1. 标题
2. 类型
3. 状态
4. 核心标签
5. 一句话摘要
6. 关系数或事件数
7. 完整度或待补录提示

### 8.2 中央卷宗主区

中央主区应是“案卷阅读面”，而不是单纯输入框区域。
建议模块顺序如下：

1. 卷宗封面
2. 核心摘要
3. 结构化属性
4. 时间线
5. 关系档案
6. 参与事件
7. 证据与引用
8. 分析与完整度
9. 备注与待确认事项

### 8.3 右侧检查器

检查器承载高频编辑，但不替代正文的阅读和判断功能。
适合放在检查器里的内容：

1. Core Fields
2. Additional Fields
3. Reference Fields
4. 快速关系草拟
5. 当前对象的局部分析摘要

### 8.4 辅助区

辅助区用于承载高频校验与批量推进能力：

1. 审校队列
2. 缺失字段提示
3. 引用来源列表
4. 与其他对象的快速对照
5. 完整度或异常预警入口

## 9. 关键交互链路

卷宗至少要支持以下链路：

1. 从对象库切换对象，正文与检查器同步更新
2. 在正文中打开引用目标卷宗，并支持返回上一卷
3. 在正文中打开独立关系卷宗，查看关系语境
4. 在正文中查看出场记录并追到事件卷宗
5. 在检查器中直接编辑对象字段与引用字段
6. 在检查器中直接新建关系对象
7. 从审校队列快速打开待补录对象
8. 从完整度或异常提示跳回缺失或冲突位置

## 10. BaseLab 独立验证页的当前结构

在 `WBS 4.1.2 / 4.1.3 / 4.1.4` 阶段，卷宗必须先在 `BaseLab` 中独立成立。

当前独立验证页至少分为三层：

1. `设计说明`
   - 解释卷宗为什么存在、如何解耦、统一模型是什么
2. `样例规范`
   - 直接展示 `objects/*.json`、`views/*.json` 样例结构
3. `工作面验证`
   - 承载对象库、卷宗正文、检查器、审校队列与关系/出场回查

默认应先落到 `设计说明`，而不是直接掉进操作区。

## 11. 当前阶段的实现策略

当前策略仍然是两阶段：

1. `WBS 4.1.2 / 4.1.3 / 4.1.4`
   - 先在 `BaseLab + 样例项目 + 模拟状态` 中验证卷宗成立性
2. `WBS 4.1.5`
   - 再把已验证过的卷宗工作面正式融合进主工作台

这样做的意义在于：

1. 不被主工作台壳层干扰判断
2. 在真实数据不完整时先验证工作面逻辑
3. 把“工作面成立性验证”和“主工作台融合”拆成两个可追踪节点

## 12. 设计约束

1. 卷宗必须优先保证事实密度和回查效率，不能为了美观把信息切得过散。
2. 卷宗强调“档案感”，但不能变成难读的拟物界面。
3. 卷宗统一模型是观察模型，不等于要求存在一个大而全的卷宗真源文件。
4. 章节不是建模前提；章节切片只是观察和输出维度。
5. 时间线、关系图、完整度、异常结果都必须区分真源与派生，不得混写。
6. AI 抽取和理解结果不得伪装成正式对象字段，必须先被外部智能体规格化。
7. `BaseLab` 中的独立验证页必须使用与正式工作台一致的对象语义和数据结构，不能做成无法回接主工作台的临时演示页。
8. 主工作台融合必须被视为独立节点处理，不与当前独立验证阶段混写。
