# WBS 4.1 卷宗工作面设计

## 工作面定位

`卷宗` 是对象事实管理工作面。

之所以使用“卷宗”这个名字，而不是继续使用泛化的“知识库”，是因为这个工作面的核心动作不是“浏览一些知识卡片”，而是：

1. 打开一份对象档案
2. 核对对象事实
3. 回查对象引用
4. 进行字段补录与审校

它更像案卷、人物档案和证据夹，而不是图谱或时间线。

当前阶段，`卷宗` 也是 `WBS 4` 的优先执行分支。原因不是它比其他工作面“更高级”，而是它最直接承接对象事实核对、字段补录和引用回查，是后续其他工作面的事实入口。

## 解决的问题

卷宗工作面主要解决四类问题：

1. 对象事实分散，无法集中核对
2. 对象字段缺失，无法快速补齐
3. 引用关系存在，但难以回查
4. 对象被多个视图使用时，缺少一个稳定的事实入口

因此，卷宗不是其他工作面的附属说明页，而是所有对象事实的首要观察入口。

## 当前阶段实施策略

用户已明确提出：当前真实数据积累仍不足，因此卷宗不应一开始就直接绑死在主工作台中开发。

本节点采用两阶段路线：

1. `WBS 4.1.2 / 4.1.3 / 4.1.4` 先在 `BaseLab` 中以独立页面和模拟数据完成能力、功能、样式验证
2. `WBS 4.1.5` 再把已验证过的卷宗工作面融合进主工作台

这样做的目的有三点：

1. 先验证“卷宗这个工作面本身是否成立”，而不是先被主界面框架绑住
2. 在真实数据尚不完整时，用样例项目和模拟状态提前验证交互链路
3. 把“工作面成立性验证”和“主工作台集成”拆成两个清晰节点，降低问题定位成本

## BaseLab 独立验证页结构

当前 `WBS 4.1.2 / 4.1.3 / 4.1.4` 的 `BaseLab` 页面不能只是一张可操作的内部验证台，还必须承担可审阅说明页职责。

因此独立验证页至少分为三层：

1. `设计说明`
   - 回答卷宗为什么存在、覆盖哪些对象类型、与其他工作面的边界是什么
2. `样例规范`
   - 直接展示 `objects/*.json` 与 `views/*.json` 的样例结构，让审阅者看到页面依赖什么数据
3. `工作面验证`
   - 承载对象筛选、引用跳转、回查、审校队列和检查器联动等真实交互验证

默认应先落到 `设计说明`，而不是一打开就直接进入操作区。

## 与其他工作面的边界

### 1. 卷宗负责什么

1. 对象浏览
2. 对象新建
3. 字段编辑
4. 引用回查
5. 审校与完整度观察

### 2. 卷宗不负责什么

1. 不负责节点推演
2. 不负责多轨编排
3. 不负责空间舞台组织
4. 不负责章节切片生成

换句话说，卷宗关心“事实是什么”，不关心“它如何被投影到图、轨道或舞台”。

## 数据驱动关系

### 1. 主要输入数据

卷宗主要由 `objects/*.json` 驱动，重点包括：

1. `characters.json`
2. `factions.json`
3. `locations.json`
4. `items.json`
5. `realm-systems.json`
6. `events.json`
7. `relations.json`
8. `clues.json`
9. `arcs.json`

### 2. 辅助输入数据

卷宗还需要读取视图状态，例如：

1. 当前对象类型过滤
2. 搜索关键词
3. 最近浏览对象
4. 已保存筛选器

这些状态应写入 `views/*.json`，而不是写回对象事实文件。

在独立验证阶段，允许先使用 `fixtures/projects/sample-novel/` 及其派生模拟状态作为卷宗样例数据源；但数据结构仍必须遵循正式对象真源和视图状态规范，不能为了验证方便另造一套无关数据格式。

### 3. 输出动作

卷宗会产生两类输出：

1. 对对象事实的正式修改
2. 对卷宗浏览状态的局部修改

第一类会影响全部工作面，第二类只影响卷宗自身。

## 页面结构

### 1. 左侧对象库

左侧对象库长期常驻，负责：

1. 搜索对象
2. 切换对象分类
3. 创建对象
4. 把当前对象送入中央卷宗主区

### 2. 中央卷宗主区

中央主区不是简单表单，而是一个“案卷阅读面”，至少包含：

1. 对象标题区
2. 核心字段区
3. 摘要与标签区
4. 引用回查区
5. 审校状态区

### 3. 右侧检查器

检查器负责当前对象的高频编辑项，不替代中央卷宗正文。

其作用更接近：

1. 快速改字段
2. 快速切换引用目标
3. 查看当前对象的局部属性摘要

### 4. 辅助区

卷宗可包含底部或折叠式辅助区，用于承载：

1. 审校队列
2. 引用来源列表
3. 缺失字段提示
4. 与其他对象的快速对照

### 5. 审阅入口

在独立验证阶段，还需要额外提供两个审阅入口：

1. 功能清单
2. 样例数据格式规范

这样用户才能同时审核：

1. 页面为什么这样设计
2. 页面当前覆盖哪些对象与关系
3. 页面依赖的数据结构是否与正式项目规范一致

## 关键交互

1. 从对象库切换对象，中央卷宗与检查器同步更新
2. 在卷宗中点击引用目标，应直接跳转到目标对象卷宗
3. 在检查器中编辑字段，应回写模型真源
4. 卷宗中的审校提示应能显示“缺什么”，而不是只显示“有问题”
5. 在独立验证阶段，上述关键交互应先在 `BaseLab` 独立页面中成立，再接入主工作台

## 设计约束

1. 卷宗必须优先保证事实密度和回查效率，不能为了美观把信息切得过散。
2. 卷宗应突出“档案感”，但不应做成难读的拟物界面。
3. 任何与时间、空间、逻辑编排有关的表达都只能作为引用结果出现，不能喧宾夺主。
4. 卷宗页面改动对象时，必须走模型写回链路；切换筛选和浏览状态时，只改卷宗视图状态。
5. `BaseLab` 中的独立验证页必须使用与正式工作台一致的对象语义、控制动作和样例数据结构，不能做成无法回接主工作台的临时演示页。
6. 主工作台融合应被视为独立节点处理，不与 `BaseLab` 阶段混写。
